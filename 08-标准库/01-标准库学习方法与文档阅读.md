# 01: 标准库学习方法与文档阅读

这篇解决两个问题:遇到任务时**知道该查哪个包**,打开一个包时**知道先看什么、后看什么**。学完你就不会再"每个函数都去搜教程",而是能自己读文档、自己跑示例。

先校准一个预期:在 JS 里,读文件、解析参数、写日志……第一反应往往是 `npm install` 找库;Go 反过来——**标准库自带电池**,日常任务八成不用装第三方包。所以"学 Go"有一半是"学标准库"。

---

## 2. 查文档的两个入口

**入口一:终端里的 `go doc`**——不用离开编辑器就能查:

```bash
go doc io.Reader
```

```text
package io // import "io"

type Reader interface {
	Read(p []byte) (n int, err error)
}
    Reader is the interface that wraps the basic Read method.

    Read reads up to len(p) bytes into p. It returns the number of bytes read (0
    <= n <= len(p)) and any error encountered. ...
```

(输出较长,后面还有几段行为约定和相关函数,这里截取了开头。)

**入口二:网页版 [pkg.go.dev](https://pkg.go.dev)**——搜包名就能看到完整文档和可运行的 Example。

拿到一个陌生包,按这个顺序读:

1. 包解决什么问题(包级说明第一段)。
2. 最小示例怎么写(Example)。
3. 核心类型有哪些(通常一两个)。
4. 常用函数出错时返回什么。

比如 `io` 包,抓住两个核心接口就抓住了主干:

```go
type Reader interface {
    Read(p []byte) (n int, err error)
}

type Writer interface {
    Write(p []byte) (n int, err error)
}
```

理解这两个接口,后面大量标准库 API 都会变得顺——下一篇专门讲它们。

## 3. 函数签名会说话

Go 的函数签名信息密度很高,看懂签名经常就不用看正文了:

```bash
go doc os.ReadFile
```

```text
package os // import "os"

func ReadFile(name string) ([]byte, error)
    ReadFile reads the named file and returns the contents. A successful call
    returns err == nil, not err == EOF. Because ReadFile reads the whole file,
    it does not treat an EOF from Read as an error to be reported.
```

一行签名能读出:传文件名(字符串)、返回全部内容(字节切片)加一个 error——说明它是**一次性读完**,不是流式。

再看一个:

```go
func Copy(dst Writer, src Reader) (written int64, err error)
```

参数是**接口**不是具体类型——说明 `io.Copy` 不关心两端是文件、网络连接还是内存 buffer,能读能写就行。

读签名时问自己四个问题:

- 参数是具体类型还是接口?
- 返回值有没有 error?
- 调用方需不需要关闭资源(返回值里有没有 `Closer`)?
- 数据是一次读完,还是流式处理?

## 4. 标准库到处是小接口

标准库的风格:**接口小,组合灵活**。比如:

```go
type ReadCloser interface {
    Reader
    Closer
}
```

好处很实际——一个函数只要收 `io.Reader`,你就可以喂给它任何"能读的东西"。第 05 章说过 Go 接口是隐式实现(结构性的,像 TS 的鸭子类型),在标准库里这一点被用到极致:

```go
func size(r io.Reader) int {
    data, _ := io.ReadAll(r)
    return len(data)
}

func main() {
    fmt.Println(size(strings.NewReader("hello"))) // 喂字符串

    f, _ := os.Open("config.txt")                 // 喂文件
    defer f.Close()
    fmt.Println(size(f))
}
```

```text
5
21
```

同一个 `size`,既能量字符串又能量文件——将来还能量 HTTP 响应体、压缩流。**认识接口,就是认识了标准库的"通用插座"。**

---

## 5. 错误处理不能省

标准库函数大量返回 error,而且错误信息本身就很有用:

```go
func main() {
    data, err := os.ReadFile("no-such.json")
    if err != nil {
        fmt.Println("出错了:", err)
        return
    }
    fmt.Println(string(data))
}
```

```text
出错了: open no-such.json: no such file or directory
```

对比 JS:Node 的同步 API 靠 throw,异步靠 reject;Go 全部走返回值,**编译器和 linter 都盯着你有没有接住它**。第 06 章的三板斧在标准库场景全部适用:

- 该返回就返回。
- 需要上下文时用 `%w` 包装。
- 判断特定错误用 `errors.Is` / `errors.As`(本章第 09 篇细讲)。

教学示例偶尔用 `_` 省略错误是为了聚焦,你自己写代码别学这个。

---

## 6. 资源要关闭

打开文件、网络连接、HTTP 响应体之后,通常都欠一个 `Close`:

```go
f, err := os.Open("data.txt")
if err != nil {
    return err
}
defer f.Close()
```

这就是 06-03 学过的标准套路:**先确认资源到手(err 检查之后),再 defer 预约关闭**。

另一个 06-03 的结论在标准库里同样成立:只读场景 Close 失败可忽略;写文件、压缩流、网络连接场景,`Close` 本身的错误可能意味着数据没落盘,要谨慎对待。

---

## 7. 一次性读 vs 流式读

同一个文件,两种读法。一次性:

```go
data, err := os.ReadFile("lines.txt")
if err != nil {
    return err
}
fmt.Print(string(data))
```

```text
first line
second line
third line
```

流式(一行一行过手,不整个装进内存):

```go
f, err := os.Open("lines.txt")
if err != nil {
    return err
}
defer f.Close()

scanner := bufio.NewScanner(f)
for scanner.Scan() {
    fmt.Println("处理:", scanner.Text())
}
if err := scanner.Err(); err != nil {
    return err
}
```

```text
处理: first line
处理: second line
处理: third line
```

| 读法 | 适合 | JS 对应 |
|------|------|---------|
| 一次性 `os.ReadFile` | 小文件、配置文件 | `fs.readFileSync` |
| 流式 `bufio.Scanner` | 大文件、网络流、持续输入 | `readline` / stream |

选错的代价:拿 `ReadFile` 读 10GB 日志,内存直接爆掉。**先问文件多大,再选读法。**

## 8. 包名短,调用时带包名

标准库包名普遍很短:`fmt`、`os`、`io`、`bufio`、`json`、`http`、`time`。使用时永远"包名.函数",来源一目了然:

```go
json.Marshal(v)
time.Now()
os.Open(name)
```

和 JS 的 `import { readFile } from 'fs'`(散名导入)不同,Go 不把函数名散进当前作用域——看到 `json.Marshal` 就知道它从哪来。别给标准库包起别名,除非真的撞名(第 09 篇会遇到一个:`math/rand` 和 `crypto/rand`)。

## 9. 文档里的 Example 值得亲手跑

pkg.go.dev 上的 Example 都能直接运行。学新包的高性价比动作:

1. 复制示例到本地,原样跑一遍。
2. 改一个参数,再跑。
3. 故意传错误输入,看错误长什么样。
4. 把它改造成自己的小函数。

第 3 步最容易被跳过,但**见过错误的样子,线上遇到时才认得出**。

## 10. 建立"问题 → 包"的索引

不要背 API,先建索引——遇到问题知道查哪儿,就赢了:

| 遇到的问题 | 查这个包 | Node 里你熟悉的对应物 |
|-----------|---------|----------------------|
| 读写数据 | `io`、`os`、`bufio` | `fs`、streams |
| 拼路径、找文件 | `path/filepath` | `path`、`glob` |
| 字符串、类型转换 | `strings`、`strconv` | `String.prototype.*`、`parseInt` |
| JSON | `encoding/json` | `JSON.parse` / `JSON.stringify` |
| 时间 | `time` | `Date` |
| 命令行参数 | `flag`、`os.Args` | `process.argv` |
| 日志 | `log/slog` | `console` / winston |
| 测试 | `testing`、`httptest` | jest / vitest |

## 本篇重点

- [ ] Go 标准库自带电池,日常任务大多不需要第三方包——学 Go 有一半是学标准库。
- [ ] 查文档两个入口:`go doc 包.名字`(终端)和 pkg.go.dev(网页,带可运行 Example)。
- [ ] 读函数签名问四件事:参数是接口吗、有没有 error、要不要 Close、一次性还是流式。
- [ ] 标准库靠 `io.Reader` / `io.Writer` 这类小接口通吃文件、网络、内存——认识接口就是认识"通用插座"。
- [ ] 错误要接住、资源要关闭、大文件走流式——这三个习惯贯穿本章所有篇目。
