# 优雅关闭与生命周期管理

Web 服务启动后，还要能可靠地关闭。关闭流程处理不好，可能导致请求中断、数据写入一半、后台任务泄露、数据库连接没有释放。

---

## 1. 服务生命周期

一个服务的生命周期通常包括：

1. 加载配置。
2. 初始化 logger。
3. 连接数据库。
4. 组装 service、handler、router。
5. 启动 HTTP server。
6. 接收关闭信号。
7. 停止接收新请求。
8. 等待正在处理的请求结束。
9. 关闭数据库和后台任务。

这些步骤应该在启动代码中清晰可见。

---

## 2. 配置 Server 超时

```go
server := &http.Server{
	Addr:              cfg.Addr,
	Handler:           handler,
	ReadHeaderTimeout: 5 * time.Second,
	ReadTimeout:       15 * time.Second,
	WriteTimeout:      30 * time.Second,
	IdleTimeout:       60 * time.Second,
}
```

常用超时含义：

| 字段 | 含义 |
| --- | --- |
| `ReadHeaderTimeout` | 读取请求头的最大时间 |
| `ReadTimeout` | 读取整个请求的最大时间 |
| `WriteTimeout` | 写响应的最大时间 |
| `IdleTimeout` | keep-alive 空闲连接最大等待时间 |

`ReadHeaderTimeout` 很重要，可以降低慢速请求头拖住连接的风险。上传大文件时要谨慎设置 `ReadTimeout`，否则正常上传也可能被切断。

---

## 3. 关闭信号

```go
ctx, stop := signal.NotifyContext(context.Background(), os.Interrupt, syscall.SIGTERM)
defer stop()

<-ctx.Done()
```

常见信号：

- `os.Interrupt`：本地按 `Ctrl+C`。
- `syscall.SIGTERM`：容器或进程管理器要求服务退出。

收到信号后，不要直接 `os.Exit`。应该进入关闭流程。

---

## 4. Shutdown 与 Close

`http.Server` 有两个相关方法：

| 方法 | 行为 |
| --- | --- |
| `Shutdown(ctx)` | 停止接收新连接，等待已有请求完成，受 context 超时控制 |
| `Close()` | 立即关闭所有连接 |

Web 服务优先使用 `Shutdown`：

```go
shutdownCtx, cancel := context.WithTimeout(context.Background(), 10*time.Second)
defer cancel()

if err := server.Shutdown(shutdownCtx); err != nil {
	return fmt.Errorf("shutdown server: %w", err)
}
```

如果超时还没关掉，调用方可以记录错误并让进程退出。

## 5. 统一管理关闭函数

多个资源需要关闭时，可以集中管理：

```go
package app

import (
	"context"
	"errors"
	"fmt"
)

type Closer func(context.Context) error

type Lifecycle struct {
	closers []Closer
}

func (l *Lifecycle) Add(closer Closer) {
	l.closers = append(l.closers, closer)
}

func (l *Lifecycle) Close(ctx context.Context) error {
	var joined error
	for i := len(l.closers) - 1; i >= 0; i-- {
		if err := l.closers[i](ctx); err != nil {
			joined = errors.Join(joined, fmt.Errorf("close resource %d: %w", i, err))
		}
	}
	return joined
}
```

后添加的资源先关闭，符合“先创建基础资源，再创建依赖资源；关闭时倒过来”的习惯。

## 6. 数据库关闭

`*sql.DB` 是连接池，程序退出时要关闭：

```go
lifecycle.Add(func(ctx context.Context) error {
	done := make(chan error, 1)
	go func() {
		done <- db.Close()
	}()

	select {
	case err := <-done:
		return err
	case <-ctx.Done():
		return ctx.Err()
	}
})
```

`db.Close()` 本身没有接收 context，这里用 goroutine 和 channel 包一层，方便统一放入关闭流程。

## 7. 后台任务退出

后台任务要监听 context：

```go
type Worker struct {
	logger *slog.Logger
}

func (w *Worker) Run(ctx context.Context) error {
	ticker := time.NewTicker(time.Minute)
	defer ticker.Stop()

	for {
		select {
		case <-ctx.Done():
			return ctx.Err()
		case <-ticker.C:
			if err := w.tick(ctx); err != nil {
				w.logger.Error("worker tick failed", slog.String("error", err.Error()))
			}
		}
	}
}
```

启动时：

```go
workerCtx, cancelWorker := context.WithCancel(context.Background())
workerDone := make(chan error, 1)

go func() {
	workerDone <- worker.Run(workerCtx)
}()

lifecycle.Add(func(ctx context.Context) error {
	cancelWorker()
	select {
	case err := <-workerDone:
		if errors.Is(err, context.Canceled) {
			return nil
		}
		return err
	case <-ctx.Done():
		return ctx.Err()
	}
})
```

后台任务不能只靠进程退出被动结束。它应该知道什么时候停止。

## 8. 就绪状态

服务开始关闭后，可以把 readiness 变成 false，让负载均衡停止发新流量：

```go
type Readiness struct {
	value atomic.Bool
}

func (r *Readiness) Ready() {
	r.value.Store(true)
}

func (r *Readiness) Draining() {
	r.value.Store(false)
}

func (r *Readiness) Handler(w http.ResponseWriter, req *http.Request) {
	if !r.value.Load() {
		http.Error(w, "draining", http.StatusServiceUnavailable)
		return
	}
	w.WriteHeader(http.StatusNoContent)
}
```

收到关闭信号后先进入 draining，再调用 `Shutdown`。

---

## 9. 学习检查

检查生命周期代码时，重点看：

1. server 是否配置超时。
2. 是否监听 `SIGINT` 和 `SIGTERM`。
3. 是否使用 `Shutdown`。
4. `Shutdown` 是否带超时。
5. 数据库连接池是否关闭。
6. 后台任务是否监听 context。
7. readiness 是否能表达关闭中的状态。
