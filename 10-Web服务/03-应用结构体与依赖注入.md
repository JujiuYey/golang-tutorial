# 应用结构体与依赖注入

Web 服务通常需要 logger、配置、数据库连接、业务服务、时间函数、ID 生成器等依赖。把这些依赖放进结构体，再从外部传入，代码会更容易测试和维护。

---

## 1. 避免全局依赖

全局变量的写法很方便：

```go
var logger *slog.Logger
var taskService *task.Service
```

问题是依赖来源不清楚，测试之间容易互相影响，初始化顺序也容易混乱。

更稳的方式是使用应用结构体：

```go
package app

import (
	"log/slog"
	"net/http"

	"example.com/task-api/internal/config"
	"example.com/task-api/internal/task"
)

type App struct {
	config      config.Config
	logger      *slog.Logger
	taskService *task.Service
	mux         *http.ServeMux
}
```

所有顶层依赖集中在 `App` 中，路由和启动逻辑都从这里拿依赖。

---

## 2. 用 Options 创建 App

构造函数应该检查依赖是否完整：

```go
package app

import (
	"fmt"
	"log/slog"
	"net/http"

	"example.com/task-api/internal/config"
	"example.com/task-api/internal/task"
)

type Options struct {
	Config      config.Config
	Logger      *slog.Logger
	TaskService *task.Service
}

func New(opts Options) (*App, error) {
	if opts.Logger == nil {
		return nil, fmt.Errorf("logger is required")
	}
	if opts.TaskService == nil {
		return nil, fmt.Errorf("task service is required")
	}
	if err := opts.Config.Validate(); err != nil {
		return nil, fmt.Errorf("validate config: %w", err)
	}

	app := &App{
		config:      opts.Config,
		logger:      opts.Logger,
		taskService: opts.TaskService,
		mux:         http.NewServeMux(),
	}
	if err := app.registerRoutes(); err != nil {
		return nil, err
	}
	return app, nil
}
```

构造函数做依赖校验，可以避免请求到达后才遇到 nil pointer。

---

## 3. Handler 也用构造函数

handler 持有自己需要的依赖：

```go
package task

import (
	"fmt"
	"log/slog"
)

type HTTPHandler struct {
	service *Service
	logger  *slog.Logger
}

func NewHTTPHandler(service *Service, logger *slog.Logger) (*HTTPHandler, error) {
	if service == nil {
		return nil, fmt.Errorf("task service is required")
	}
	if logger == nil {
		return nil, fmt.Errorf("logger is required")
	}
	return &HTTPHandler{
		service: service,
		logger:  logger,
	}, nil
}
```

真实文件里需要导入 `fmt`。构造函数里校验依赖，比在方法里反复判断更清楚。

## 4. 接口放在使用处

当 handler 只需要 service 的一部分能力时，可以定义一个小接口：

```go
package task

import "context"

type TaskReader interface {
	List(ctx context.Context) ([]Task, error)
	FindByID(ctx context.Context, id string) (Task, error)
}

type ReadHandler struct {
	reader TaskReader
}
```

小接口让测试替身更容易写：

```go
type fakeTaskReader struct {
	tasks []task.Task
	err   error
}

func (f fakeTaskReader) List(ctx context.Context) ([]task.Task, error) {
	if f.err != nil {
		return nil, f.err
	}
	return f.tasks, nil
}
```

接口大小应该跟使用需求匹配。为了抽象而抽象，会让代码更难读。

## 5. 注入函数依赖

有些依赖用函数就够了，例如当前时间和 ID 生成：

```go
package task

import (
	"fmt"
	"time"
)

type IDGenerator func() string
type Clock func() time.Time

type Service struct {
	repo Repository
	newID IDGenerator
	now   Clock
}

func NewService(repo Repository, newID IDGenerator, now Clock) (*Service, error) {
	if repo == nil {
		return nil, fmt.Errorf("repository is required")
	}
	if newID == nil {
		return nil, fmt.Errorf("id generator is required")
	}
	if now == nil {
		return nil, fmt.Errorf("clock is required")
	}
	return &Service{repo: repo, newID: newID, now: now}, nil
}
```

测试时可以注入固定时间：

```go
fixedNow := func() time.Time {
	return time.Date(2026, 1, 2, 3, 4, 5, 0, time.UTC)
}
```

这样测试不依赖真实时钟，断言会稳定很多。

---

## 6. 依赖注入的尺度

Go 项目里常见的手写依赖注入已经足够：

```go
repo := task.NewMemoryRepository()
service, err := task.NewService(repo, newID, time.Now)
if err != nil {
	return err
}
handler, err := task.NewHTTPHandler(service, logger)
if err != nil {
	return err
}
```

当项目变得很大时，可以再考虑专门的装配层或代码生成工具。课程阶段优先掌握手写方式，因为它能让依赖关系透明可见。

---

## 7. 学习检查

看一个 Web 服务的依赖组织时，检查：

1. 是否存在会被多个测试共享修改的全局变量。
2. 构造函数是否校验关键依赖。
3. handler 是否从外部接收 service。
4. service 是否从外部接收 repository。
5. 时间、ID、随机数这类依赖是否能替换。
6. 接口是否足够小。
7. 依赖方向是否从外层指向内层。
