# 03: 应用结构体与依赖注入

logger、配置、数据库连接、业务服务、时间函数……一个 Web 服务的依赖不少,放全局变量最省事,但测试和维护会跟着遭殃。这篇解决的问题是:**依赖怎么传,才能让每个组件既拿得到需要的东西,又能在测试里被随手替换**。学完你就掌握了 Go 圈最主流的依赖注入方式——不用框架,就是构造函数传参。

对比 Node:你可能习惯 `require('./db')` 拿单例(模块缓存就是隐式全局),测试时用 `jest.mock` 整个劫持模块。Go 的路线相反:**依赖显式地从构造函数进来**,测试时传个假的进去就行,不需要任何 mock 魔法。

---

## 1. 避免全局依赖

全局变量的写法很诱人:

```go
var logger *slog.Logger
var taskService *task.Service
```

### 🕳️ 坑:全局依赖的三宗罪

以为会怎样:哪里要用哪里拿,和 Node 的模块单例一样方便。
实际怎样:① 看函数签名完全猜不到它依赖什么;② 测试之间共享同一个全局状态,互相污染,`-race` 还可能报数据竞争;③ 初始化顺序全靠"碰巧"——谁先 init 谁说了算。
为什么:全局变量把依赖关系从"显式传递"变成了"隐式约定",代码规模一大,约定必崩。

更稳的方式是**应用结构体**:所有顶层依赖集中在一个 `App` 上:

```go
type App struct {
    config      config.Config
    logger      *slog.Logger
    taskService *task.Service
    mux         *http.ServeMux
}
```

路由注册、启动逻辑都从 `App` 身上拿依赖——第 01 篇 main 装配的就是它。

---

## 2. 用 Options 创建 App

依赖多了,构造函数参数排一排容易传错位。用 Options 结构体按名传参(效果类似 JS 的对象参数 `createApp({ logger, service })`):

```go
type Options struct {
    Config      config.Config
    Logger      *slog.Logger
    TaskService *task.Service
}

func New(opts Options) (*App, error) {
    if opts.Logger == nil {
        return nil, fmt.Errorf("logger is required")
    }
    if opts.TaskService == nil {
        return nil, fmt.Errorf("task service is required")
    }
    if err := opts.Config.Validate(); err != nil {
        return nil, fmt.Errorf("validate config: %w", err)
    }

    app := &App{
        config:      opts.Config,
        logger:      opts.Logger,
        taskService: opts.TaskService,
        mux:         http.NewServeMux(),
    }
    if err := app.registerRoutes(); err != nil {
        return nil, err
    }
    return app, nil
}
```

**构造函数当门卫**:依赖缺了当场报错,而不是等第一个请求进来才 nil pointer panic。和上一篇"配置错误死在启动时"是同一个哲学。

---

## 3. Handler 也用构造函数

同样的套路一路向下,handler 持有自己需要的依赖:

```go
type HTTPHandler struct {
    service *Service
    logger  *slog.Logger
}

func NewHTTPHandler(service *Service, logger *slog.Logger) (*HTTPHandler, error) {
    if service == nil {
        return nil, fmt.Errorf("task service is required")
    }
    if logger == nil {
        return nil, fmt.Errorf("logger is required")
    }
    return &HTTPHandler{service: service, logger: logger}, nil
}
```

在构造函数里校验一次,好过在每个方法里反复判断 nil。

**一句话总结:每一层都长一个样——struct 装依赖,NewXxx 校验并组装。**

---

## 4. 接口放在使用处

handler 常常只用到 service 的一小部分能力。这时可以定义一个**小接口**,只声明自己要的:

```go
type TaskReader interface {
    List(ctx context.Context) ([]Task, error)
    FindByID(ctx context.Context, id string) (Task, error)
}

type ReadHandler struct {
    reader TaskReader
}
```

小接口的直接回报是测试替身好写——两个方法随手就能 fake:

```go
type fakeTaskReader struct {
    tasks []task.Task
    err   error
}

func (f fakeTaskReader) List(ctx context.Context) ([]task.Task, error) {
    if f.err != nil {
        return nil, f.err
    }
    return f.tasks, nil
}
```

这就是模块 05 说过的"接口越小越好"落到 Web 服务里的样子。对比 JS:这里的 fake 相当于 `jest.fn()` 手搓版——但不需要 mock 框架,因为 Go 接口是隐式实现的,方法对上就算数。

注意尺度:接口大小跟**使用需求**匹配。为了抽象而抽象,只会让读代码的人多跳一层。

## 5. 注入函数依赖

有些依赖不用大动干戈定义接口,一个**函数类型**就够了——典型的是时间和 ID 生成:

```go
type IDGenerator func() string
type Clock func() time.Time

type Service struct {
    repo  Repository
    newID IDGenerator
    now   Clock
}

func NewService(repo Repository, newID IDGenerator, now Clock) (*Service, error) {
    if repo == nil {
        return nil, fmt.Errorf("repository is required")
    }
    if newID == nil {
        return nil, fmt.Errorf("id generator is required")
    }
    if now == nil {
        return nil, fmt.Errorf("clock is required")
    }
    return &Service{repo: repo, newID: newID, now: now}, nil
}
```

生产环境传 `time.Now`,测试传固定时间。跑一段真实代码看看效果:

```go
func main() {
    // 依赖不完整:构造函数当场报错
    _, err := task.NewService(nil, nil, nil)
    fmt.Println("缺依赖:", err)

    // 注入固定时间和固定 ID:输出完全可预测
    repo := task.NewMemoryRepository()
    fixedNow := func() time.Time {
        return time.Date(2026, 1, 2, 3, 4, 5, 0, time.UTC)
    }
    newID := func() string { return "task_1" }

    service, _ := task.NewService(repo, newID, fixedNow)
    created, _ := service.Create(context.Background(), task.CreateInput{Title: "  learn di  "})
    fmt.Printf("created: id=%s title=%q at=%s\n", created.ID, created.Title, created.CreatedAt.Format(time.RFC3339))
}
```

```text
缺依赖: repository is required
created: id=task_1 title="learn di" at=2026-01-02T03:04:05Z
```

时间和 ID 都被钉死,断言可以精确到每个字符——这就是"把不确定性注入成依赖"的价值。直接在 service 里写 `time.Now()`,这种测试就写不出来。

## 6. 依赖注入的尺度

Go 项目里,手写装配就是主流的依赖注入:

```go
repo := task.NewMemoryRepository()
service, err := task.NewService(repo, newID, time.Now)
if err != nil {
    return err
}
handler, err := task.NewHTTPHandler(service, logger)
if err != nil {
    return err
}
```

对比 Node:NestJS 用装饰器 + IoC 容器自动装配,Go 社区绝大多数项目就是像上面这样一行行手写(依赖特别多的大型项目才会考虑 wire 这类代码生成工具)。手写的好处是**依赖关系全在明面上**——读 main 就是读整个服务的架构图。

**一句话总结:Go 的依赖注入没有魔法,就是"构造函数传参 + main 里手动装配"。**

---

## 本篇重点

- [ ] 别用全局变量装依赖:签名看不出依赖、测试互相污染、初始化顺序失控。
- [ ] 套路统一:struct 装依赖 + `NewXxx` 构造函数校验必填项,缺了启动时就报错。
- [ ] 依赖多时用 Options 结构体按名传参(= JS 的对象参数)。
- [ ] 接口定义在使用方、按需求裁小;时间、ID 这类不确定性用函数类型注入,测试传固定值。
- [ ] Go 主流依赖注入 = main 里手写装配,没有容器、没有 mock 框架。

## 练习

给 `book-api` 补上依赖注入:

要求:

1. `book.Service` 依赖 `Repository` 接口、`IDGenerator`、`Clock`,构造函数校验三者非 nil。
2. `Service.Create` 用注入的 clock 填 `CreatedAt`,用注入的 id 生成器填 `ID`。
3. 写一个测试:注入固定时间 `2026-03-01T00:00:00Z` 和固定 ID `book_1`,断言创建结果的每个字段。
4. 故意传 nil repo,断言构造函数返回错误。

提示:固定时钟就是 `func() time.Time { return time.Date(...) }`;测试里不需要任何 mock 库,手写 fake 或直接用内存 repository 都行。
