# 11: 常见并发模式

并发模式可以理解为常见的 goroutine 和 channel 组织方式。本节讲 worker pool、fan-out/fan-in 和 pipeline。

---

## 1. worker pool

worker pool 用固定数量的 worker 处理一批任务，避免为每个任务无限制启动 goroutine。

```go
func worker(id int, jobs <-chan Job, results chan<- Result) {
    for job := range jobs {
        results <- handle(job)
    }
}
```

启动 worker：

```go
jobs := make(chan Job)
results := make(chan Result)

for i := 0; i < 3; i++ {
    go worker(i, jobs, results)
}
```

---

## 2. worker pool 完整结构

```go
var wg sync.WaitGroup

jobs := make(chan Job)
results := make(chan Result)

for i := 0; i < workerCount; i++ {
    wg.Add(1)
    go func(id int) {
        defer wg.Done()
        for job := range jobs {
            results <- handle(job)
        }
    }(i)
}

go func() {
    wg.Wait()
    close(results)
}()

go func() {
    defer close(jobs)
    for _, job := range allJobs {
        jobs <- job
    }
}()

for result := range results {
    fmt.Println(result)
}
```

关键点：

- 生产者关闭 `jobs`。
- worker 退出后，协调者关闭 `results`。
- 主 goroutine range `results`。

---

## 3. fan-out

fan-out 表示多个 goroutine 从同一个输入读取任务。

```go
for i := 0; i < workerCount; i++ {
    go worker(i, jobs, results)
}
```

一个 `jobs` channel，多个 worker 接收。每个任务只会被一个 worker 处理。

---

## 4. fan-in

fan-in 表示把多个输入合并成一个输出。

```go
func merge[T any](inputs ...<-chan T) <-chan T {
    out := make(chan T)
    var wg sync.WaitGroup

    for _, ch := range inputs {
        wg.Add(1)
        go func(ch <-chan T) {
            defer wg.Done()
            for v := range ch {
                out <- v
            }
        }(ch)
    }

    go func() {
        wg.Wait()
        close(out)
    }()

    return out
}
```

每个输入 channel 由一个 goroutine 转发到 `out`。所有输入结束后关闭 `out`。

---

## 5. pipeline

pipeline 把处理拆成多个阶段，每个阶段通过 channel 连接。

```go
func gen(nums ...int) <-chan int {
    out := make(chan int)
    go func() {
        defer close(out)
        for _, n := range nums {
            out <- n
        }
    }()
    return out
}

func square(in <-chan int) <-chan int {
    out := make(chan int)
    go func() {
        defer close(out)
        for n := range in {
            out <- n * n
        }
    }()
    return out
}
```

使用：

```go
for v := range square(gen(1, 2, 3)) {
    fmt.Println(v)
}
```

---

## 6. pipeline 的取消

如果下游提前退出，上游可能卡在发送上。带 context 的版本更安全：

```go
func square(ctx context.Context, in <-chan int) <-chan int {
    out := make(chan int)
    go func() {
        defer close(out)
        for n := range in {
            select {
            case out <- n * n:
            case <-ctx.Done():
                return
            }
        }
    }()
    return out
}
```

每个阶段都要响应取消。

---

## 7. 限制并发

用缓冲 channel 作为信号量：

```go
sem := make(chan struct{}, limit)

for _, item := range items {
    sem <- struct{}{}
    wg.Add(1)
    go func(item Item) {
        defer wg.Done()
        defer func() { <-sem }()
        process(item)
    }(item)
}

wg.Wait()
```

容量 `limit` 控制同时运行的 goroutine 数量。

---

## 8. 错误传播

结果类型可以同时包含值和错误：

```go
type Result[T any] struct {
    Value T
    Err   error
}
```

worker 发送 `Result[T]`，接收方统一处理错误。

如果一个错误应该取消整个任务组，配合 context 取消。

---

## 9. 结果顺序

并发处理通常不保证结果顺序。

如果需要保持输入顺序，可以给任务带上索引：

```go
type Job struct {
    Index int
    URL   string
}

type Result struct {
    Index int
    Body  []byte
    Err   error
}
```

收集后按 `Index` 放回结果切片。

---

## 10. 模式选择

- 任务数量大，需要限制并发：worker pool。
- 多个来源合并：fan-in。
- 一个来源分给多个处理者：fan-out。
- 多阶段流式处理：pipeline。
- 下游可能提前退出：加 context。
- 需要稳定顺序：带索引收集。

模式要服务清晰的生命周期，而不是增加并发结构的复杂度。
