# 04: 路由、路径参数与查询参数

这篇解决"Express 的路由功能在 Go 里怎么办"：`app.get('/users/:id', ...)` 这种带方法、带路径参数的路由，标准库 `ServeMux`（Go 1.22 起）已经原生支持，不需要急着找框架。学完你能注册带方法的路由、取路径参数并校验，还知道旧版 Go 的手动解法长什么样。

---

## 1. 基本路由：pattern 里直接写方法

Go 1.22 起，pattern 可以带 HTTP 方法，写法是**"方法 空格 路径"**：

```go
mux := http.NewServeMux()

mux.HandleFunc("GET /health", health)
mux.HandleFunc("GET /users", listUsers)
mux.HandleFunc("POST /users", createUser)
```

对照 Express：

```text
Express:  app.get('/users', listUsers)   app.post('/users', createUser)
Go:       "GET /users"                    "POST /users"    ← 方法写进 pattern 字符串
```

```bash
curl http://localhost:18011/health
```

```text
ok
```

方法对不上时，mux **自动**替你返回 405：

```bash
curl -i -X POST http://localhost:18011/health
```

```text
HTTP/1.1 405 Method Not Allowed
Allow: GET, HEAD
Content-Type: text/plain; charset=utf-8
X-Content-Type-Options: nosniff
Content-Length: 19

Method Not Allowed
```

连 `Allow` 头都帮你带上了（注册 GET 会自动附送 HEAD）——上一篇手写的方法检查，在新 mux 里免费获得。旧版本 Go 没有这个能力，就得用第 5 节的手动分发。

---

## 2. 路径参数

Express 的 `'/users/:id'` + `req.params.id`，Go 写成 `{id}` + `r.PathValue("id")`：

```go
mux.HandleFunc("GET /users/{id}", getUser)

func getUser(w http.ResponseWriter, r *http.Request) {
	id, err := strconv.Atoi(r.PathValue("id"))
	if err != nil {
		http.Error(w, "invalid id", http.StatusBadRequest)
		return
	}
	fmt.Fprintf(w, "user #%d\n", id)
}
```

```bash
curl http://localhost:18011/users/42
curl -i http://localhost:18011/users/abc
```

```text
user #42
```

```text
HTTP/1.1 400 Bad Request
Content-Type: text/plain; charset=utf-8
X-Content-Type-Options: nosniff
Content-Length: 11

invalid id
```

和查询参数一个道理：`PathValue` 返回的**永远是字符串**，`{id}` pattern 不会帮你保证它是数字。`/users/abc` 一样能匹配进来，所以 `strconv.Atoi` + 400 这两行是标配，不是可选项。

---

## 3. 旧版本：手动解析路径

Go 1.22 之前没有 `{id}`，得自己拆路径——读老代码一定会遇到这个模式：

```go
func legacyGetUser(w http.ResponseWriter, r *http.Request) {
	const prefix = "/legacy/users/"
	if !strings.HasPrefix(r.URL.Path, prefix) {
		http.NotFound(w, r)
		return
	}
	id := strings.TrimPrefix(r.URL.Path, prefix)
	if id == "" || strings.Contains(id, "/") {
		http.Error(w, "invalid id", http.StatusBadRequest)
		return
	}
	fmt.Fprintf(w, "legacy user %s\n", id)
}
```

```bash
curl http://localhost:18011/legacy/users/7
curl -i http://localhost:18011/legacy/users/
```

```text
legacy user 7
```

```text
HTTP/1.1 400 Bad Request
Content-Type: text/plain; charset=utf-8
X-Content-Type-Options: nosniff
Content-Length: 11

invalid id
```

注意手动解析要自己防三件事：**空 ID**（`/legacy/users/` 剪完前缀剩空串）、**多余路径段**（`/legacy/users/7/posts` 剪完剩 `7/posts`）、URL 转义。新 pattern 把这些都处理了——能用 1.22+ 就用 1.22+。

## 4. 路径参数 vs 查询参数

两种参数各管一摊，选错了不报错但 API 会很别扭：

| | 路径参数 | 查询参数 |
|---|---------|---------|
| 长相 | `GET /users/42` | `GET /users?page=2&q=alice` |
| 语义 | **资源是谁**（标识） | **怎么看这个资源**（过滤、分页、排序） |
| 缺了会怎样 | 路由不匹配（404） | 正常匹配，值为空串 |

读取查询参数上一篇讲过（`r.URL.Query().Get("page")`），这里只补一条经验：**能想象"这东西有自己的详情页"就用路径参数，只是筛选条件就用查询参数。**

## 5. Method 分发：switch 写法

不用 method pattern 时（旧 Go 版本，或想把同路径的方法集中一处），在 handler 里 switch：

```go
func users(w http.ResponseWriter, r *http.Request) {
	switch r.Method {
	case http.MethodGet:
		listUsers(w, r)
	case http.MethodPost:
		createUser(w, r)
	default:
		w.Header().Set("Allow", "GET, POST")
		http.Error(w, "method not allowed", http.StatusMethodNotAllowed)
	}
}
```

```bash
curl -i -X PUT http://localhost:18011/v0/users
```

```text
HTTP/1.1 405 Method Not Allowed
Allow: GET, POST
Content-Type: text/plain; charset=utf-8
X-Content-Type-Options: nosniff
Content-Length: 19

method not allowed
```

这就是手动版的第 1 节：`default` 分支负责 405 + `Allow` 头，一个都不能省。（switch 不写 `default` 的话，不支持的方法会**静默返回 200 空响应**——比报错还难查。）

## 6. NotFound

资源不存在时：

```go
http.NotFound(w, r) // 等价于 http.Error(w, "404 page not found", 404)
```

"路由没匹配上"和"路由匹配了但 ID 查不到数据"都是 404，语义上没毛病。JSON API 里通常换成自己的 JSON 格式 404（第 07 篇的 `writeError`），保持错误格式统一。

---

## 7. 路由顺序和匹配规则

同一个请求可能被多个 pattern 罩住，比如同时注册了 `/users/{id}` 和 `/users/me`。ServeMux 的规则是**更具体的赢**，和注册顺序无关：

```go
mux.HandleFunc("GET /users/{id}", getUser)
mux.HandleFunc("GET /users/me", getMe) // 更具体
```

```bash
curl http://localhost:18011/users/me
```

```text
current user
```

`/users/me` 进了 `getMe`，没被 `{id}` 截胡（否则第 2 节的 Atoi 会把它打成 400）。这比 Express 舒服——Express 按注册顺序匹配，`/users/me` 写在 `:id` 后面就永远进不去。

设计路由本身的建议：

- 资源用复数名词：`/users`。
- 子资源表达层级：`/users/{id}/posts`。
- 动作交给 method 表达（POST 创建、DELETE 删除），不要发明 `/users/create` 这种路径。

---

## 8. 路由分组

标准库没有 Express 的 `Router` 那种分组，但一个普通函数就够用：

```go
func registerUserRoutes(mux *http.ServeMux, h UserHandler) {
	mux.HandleFunc("GET /users", h.List)
	mux.HandleFunc("POST /users", h.Create)
	mux.HandleFunc("GET /users/{id}", h.Get)
}
```

main 里一行 `registerUserRoutes(mux, h)`，每类资源一个注册函数，路由表清清爽爽。**Go 的哲学：框架给的"特性"，很多时候一个函数就能替。**

## 9. 版本前缀

```text
/api/v1/users
```

API 版本可以放路径前缀，也可以放 header 里管理。课程阶段用路径前缀——最直观，curl 一眼能看出版本。

## 10. 路由设计检查

1. 路径是否表达资源（名词），方法是否表达动作（动词）？
2. 路径参数 ID 是否做了格式校验（`Atoi` + 400）？
3. 方法不支持时是否返回 405 + `Allow` 头（新 pattern 免费送，switch 写法要自己写）？
4. 有没有会互相截胡的模糊路径？
5. 路由注册是否集中在一处，一眼能看全？

---

## 本篇重点

- [ ] Go 1.22+ 的 mux pattern 支持 `"GET /users/{id}"`：方法写进字符串，方法不匹配自动 405 + `Allow` 头。
- [ ] `r.PathValue("id")` ≈ Express 的 `req.params.id`，但返回的是字符串——`strconv.Atoi` + 400 是标配。
- [ ] 路径参数表达"资源是谁"，查询参数表达"怎么筛"；分不清就想"它有没有详情页"。
- [ ] 匹配规则是"更具体的赢"，与注册顺序无关（和 Express 按顺序匹配不同）。
- [ ] switch 分发方法时 `default` 必须返回 405，否则未知方法静默 200。

## 练习

1. 注册 `GET /health`，并用 `curl -i -X POST` 验证自动 405。
2. 注册 `GET /users` 和 `POST /users`。
3. 注册 `GET /users/{id}`，解析整数 ID，非法 ID 返回 400。
4. 写旧版本兼容的 `/users/` 前缀解析（提示：`TrimPrefix` 之后记得防空 ID 和多余路径段）。
5. 用 switch 分发方法，对不支持的方法返回 405 和 `Allow` header。
