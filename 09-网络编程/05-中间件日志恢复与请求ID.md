# 05: 中间件、日志、恢复与请求 ID

这篇解决"Express 的 `app.use(...)` 在 Go 里怎么写"：日志、鉴权、panic 恢复、请求 ID、CORS 这些每个请求都要过一遍的横切逻辑，不该复制进每个 handler。Go 没有中间件"机制"——它只是第 02 篇那个 Handler 接口的一个玩法：**吃进一个 Handler，吐出一个包了壳的 Handler**。

---

## 1. 中间件形态

```go
func Middleware(next http.Handler) http.Handler {
	return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
		// before：请求进来先干点事
		next.ServeHTTP(w, r)
		// after：里面的干完了再干点事
	})
}
```

拆开看这个签名：

```text
func Middleware(next http.Handler) http.Handler
//              ──────┬──────────  ──────┬─────
//        ①收一个"里层"handler    ②返回一个新 handler(包了一层壳)
```

对比 Express：那边中间件是 `(req, res, next) => { ...; next() }`，**运行时**靠调 `next()` 放行；Go 这边是**函数包函数**，组装时就把洋葱一层层裹好。还有个 Express 要靠 `res.on('finish')` 才能做的事，Go 白送：`next.ServeHTTP` 返回后接着写代码，就是"响应完成后"的钩子——算耗时就是这么实现的。

**一句话总结：中间件 = 输入 Handler、输出 Handler 的函数，没有更多魔法。**

---

## 2. 日志中间件

```go
func Logger(next http.Handler) http.Handler {
	return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
		start := time.Now()
		next.ServeHTTP(w, r)
		slog.Info("request",
			"method", r.Method,
			"path", r.URL.Path,
			"duration_ms", time.Since(start).Milliseconds(),
		)
	})
}
```

用 `Logger(hello)` 包住一个耗时 30ms 的 handler，curl 一下，服务端日志：

```text
2026/07/16 10:22:30 INFO request method=GET path=/hello duration_ms=30
```

`next.ServeHTTP` 前记时间、后算差值——第 1 节的 before/after 各占一行。但这个版本有个缺口：**拿不到响应状态码**。`w` 只有写入方法没有"读回已写状态"的方法，所以要用下一节的小技巧。

---

## 3. 记录状态码：包装 ResponseWriter

思路：自己造一个 ResponseWriter，把真的嵌进来，**拦截** `WriteHeader` 顺手记下状态码：

```go
type statusWriter struct {
	http.ResponseWriter // 嵌入:自动继承所有方法
	status int
}

func (w *statusWriter) WriteHeader(status int) {
	w.status = status                  // ①先记下来
	w.ResponseWriter.WriteHeader(status) // ②再交给真的去发
}
```

这是结构体嵌入的经典应用：只重写 `WriteHeader` 一个方法，`Write`、`Header` 自动透传给里面那个真 writer。日志中间件里换上它：

```go
sw := &statusWriter{ResponseWriter: w, status: http.StatusOK}
next.ServeHTTP(sw, r) // 里层 handler 用的是我们的"间谍"writer
```

请求一个返回 201 的接口，日志里状态码有了：

```text
2026/07/16 10:22:30 INFO request method=GET path=/create status=201 duration_ms=0
```

🕳️ 坑：初始值必须设 `http.StatusOK`。以为 handler 总会调 `WriteHeader` → 实际大多数 handler 直接写 body（隐式 200），`WriteHeader` 根本没被调 → 不设初始值就会记出 `status=0`。

## 4. panic 恢复

handler 里一个数组越界，默认会打爆整条连接。用 06 模块学的 `defer` + `recover` 做个安全网：

```go
func Recover(next http.Handler) http.Handler {
	return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
		defer func() {
			if rec := recover(); rec != nil {
				slog.Error("panic", "value", rec)
				http.Error(w, "internal server error", http.StatusInternalServerError)
			}
		}()
		next.ServeHTTP(w, r)
	})
}
```

包住一个必炸的 handler（`users[3]`，切片是空的）试试：

```bash
curl -i http://localhost:18012/boom
```

```text
HTTP/1.1 500 Internal Server Error
Content-Type: text/plain; charset=utf-8
X-Content-Type-Options: nosniff
Content-Length: 22

internal server error
```

服务端日志：

```text
2026/07/16 10:22:30 ERROR panic value="runtime error: index out of range [3] with length 0"
```

客户端得到体面的 500，服务活着继续接下一个请求。（说明：Go 的 http 服务器本身也会 recover 单个请求的 panic，进程不会真的挂——但它只往 stderr 甩堆栈，响应直接断掉。自己写 Recover 是为了**统一记日志 + 给客户端一个正常的 500**。）注意：panic 恢复是兜底，不是错误处理——业务错误还是走 error 返回值。

## 5. 请求 ID

排查问题时要能把一个请求在各处日志里串起来，办法是给每个请求发个"身份证"：

```go
type requestIDKey struct{}

func RequestID(next http.Handler) http.Handler {
	return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
		id := r.Header.Get("X-Request-ID")
		if id == "" {
			id = generateID() // 上游没给就自己造一个
		}
		ctx := context.WithValue(r.Context(), requestIDKey{}, id)
		w.Header().Set("X-Request-ID", id)
		next.ServeHTTP(w, r.WithContext(ctx))
	})
}
```

```bash
curl -i http://localhost:18012/whoami
```

```text
HTTP/1.1 200 OK
X-Request-Id: f5b6bd407aba
Content-Length: 25
Content-Type: text/plain; charset=utf-8

request id: f5b6bd407aba
```

上游带了 ID 就沿用（跨服务追踪就靠这个）：

```bash
curl -H 'X-Request-ID: trace-42' http://localhost:18012/whoami
```

```text
request id: trace-42
```

两个细节：key 用私有空结构体类型 `requestIDKey{}` 而不是字符串，避免不同包撞 key；**context value 只放请求元数据**（request id、登录用户），别拿它偷运业务参数——那是函数参数的活。

## 6. 鉴权中间件

"提前拦截"型中间件：条件不满足就自己写响应、**不调 next**：

```go
func Auth(next http.Handler) http.Handler {
	return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
		token := r.Header.Get("Authorization")
		if token == "" {
			http.Error(w, "unauthorized", http.StatusUnauthorized)
			return // 拦下,里层 handler 根本不会执行
		}
		next.ServeHTTP(w, r)
	})
}
```

```bash
curl -i http://localhost:18012/secret
```

```text
HTTP/1.1 401 Unauthorized
Content-Type: text/plain; charset=utf-8
X-Content-Type-Options: nosniff
Content-Length: 13

unauthorized
```

```bash
curl -H 'Authorization: Bearer abc' http://localhost:18012/secret
```

```text
top secret
```

真实鉴权还要验签名、过期时间、权限范围——但中间件的骨架就是这样：**要么放行（调 next），要么拦截（写响应 + return），二选一，别两头都沾。**

## 7. 链式组合：洋葱模型

多个中间件套娃：

```go
handler := Recover(Logger(Auth(mux)))
```

执行顺序从外到内再从里到外，像穿过洋葱。用两个会打印的中间件实证：

```go
func mw(name string, next http.Handler) http.Handler {
	return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
		fmt.Println(name, "进入")
		next.ServeHTTP(w, r)
		fmt.Println(name, "离开")
	})
}

// mux.Handle("/order", mw("A", mw("B", inner)))
```

curl 一次，服务端打印：

```text
A 进入
B 进入
handler 执行
B 离开
A 离开
```

```text
请求 → A → B → handler
              ↓
响应 ← A ← B ←┘
```

顺序影响行为：**Recover 放最外层**（里面任何一层炸了都接得住）、日志尽量靠外（拦截的请求也想记录）、鉴权靠内（让日志能记下 401）。

## 8. CORS 简介

CORS 是浏览器的跨域安全机制（前端老朋友了）。服务端的活就是回应答头 + 处理预检请求：

```go
func CORS(next http.Handler) http.Handler {
	return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
		w.Header().Set("Access-Control-Allow-Origin", "https://example.com")
		w.Header().Set("Access-Control-Allow-Methods", "GET, POST, OPTIONS")
		if r.Method == http.MethodOptions {
			w.WriteHeader(http.StatusNoContent) // 预检到此为止
			return
		}
		next.ServeHTTP(w, r)
	})
}
```

```bash
curl -i -X OPTIONS http://localhost:18012/cors
```

```text
HTTP/1.1 204 No Content
Access-Control-Allow-Methods: GET, POST, OPTIONS
Access-Control-Allow-Origin: https://example.com
```

🕳️ 坑：`Allow-Origin: *` 图省事 → 任何网站都能带用户身份调你的接口 → 涉及凭证和敏感接口时严格白名单，别用 `*`。

---

## 9. 中间件不要吞掉响应

两条纪律：

- **提前拦截时把响应写完整**（状态码 + body + `return`）。只 `return` 不写响应，客户端会收到一个莫名其妙的 200 空响应。
- **调了 `next.ServeHTTP` 之后别再改状态码**。里层多半已经把状态码发出去了，再改就是第 02 篇的 `superfluous WriteHeader`。

---

## 10. 中间件设计检查

1. 是否只处理横切逻辑（日志/鉴权/恢复），没夹带业务？
2. 每条路径是否明确"放行"或"拦截"，拦截时响应写完整了吗？
3. 换 context 时是否用 `r.WithContext(ctx)` 传下去了？
4. 组合顺序合理吗（Recover 最外、Logger 靠外、Auth 靠内）？
5. 包装 ResponseWriter 时初始 status 设了 200 吗？

---

## 本篇重点

- [ ] 中间件就是 `func(next http.Handler) http.Handler`：组装时裹洋葱，不像 Express 靠运行时 `next()`。
- [ ] `next.ServeHTTP` 之前是 before、之后是 after——after 白送"响应完成"钩子，记耗时全靠它。
- [ ] 想知道状态码就嵌入包装 ResponseWriter 拦截 `WriteHeader`，初始值必须 200。
- [ ] Recover = defer + recover 兜底,放最外层；context value 只放请求元数据，key 用私有类型。
- [ ] 拦截型中间件：写完整响应 + return；调过 next 之后不要再动状态码。

2. 包装 `ResponseWriter` 记录 status，并入日志（提示：结构体嵌入 + 只重写 `WriteHeader`）。
3. 写 `Recover` 中间件捕获 panic 并返回 500（提示：故意写一个越界访问来验证）。
