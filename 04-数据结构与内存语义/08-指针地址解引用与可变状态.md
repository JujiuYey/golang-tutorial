# 08: 指针：地址、解引用与可变状态

这篇把前面反复露脸的 `*` 和 `&` 正式讲透，并且拆掉一个流传极广的误解：**"值类型放栈上、引用类型放堆上"——错，这是两个不相干的问题**。Go 的指针比 C 温和得多：能取地址、解引用、判 nil，不能做算术；至于变量住栈还是堆，是编译器逃逸分析说了算，本篇用编译器自己的输出当证据。

JS 里没有指针这个东西，但你天天在用它的效果：JS 变量存对象时存的就是"引用"。Go 只是把这层从暗处搬到明处，让你自己决定"给你一份复制"还是"给你原件的地址"。

---

## 1. 取地址

`&x` 读作"x 的地址"：

```go
x := 10
p := &x

fmt.Println(p)
fmt.Println(*p)
fmt.Printf("%T\n", p)
```

```text
0x14000092030
10
*int
```

（第一行是内存地址，每次运行、每台机器都不同。）`p` 的类型是 `*int`——"指向 int 的指针"。星号在类型里表示"指针类型"，在表达式里表示"解引用"，两个位置两种含义：

```text
var p *int        *p = 20
//    ─┬──        ┬
//  类型：指向int的指针   操作：顺着指针找到那个int
```

---

## 2. 解引用

`*p` 是"顺着地址找到本体"。通过它读，也通过它写：

```go
x := 10
p := &x

*p = 20

fmt.Println(x)
```

```text
20
```

改 `*p`，`x` 变了——因为 `*p` 就是 `x` 本人，不是副本。

---

## 3. 指针的零值是 nil

```go
var p *int
fmt.Println(p == nil)
```

```text
true
```

解引用 nil 指针 = 03 模块见过的经典 panic：

```go
package main

import "fmt"

func main() {
	var p *int
	fmt.Println(*p)
}
```

```text
panic: runtime error: invalid memory address or nil pointer dereference
[signal SIGSEGV: segmentation violation code=0x2 addr=0x0 pc=0x100819470]

goroutine 1 [running]:
main.main()
	./main.go:7 +0x20
exit status 2
```

这就是 Go 版的 "cannot read property of null"。用指针前先确认非 nil。

---

## 4. new

`new(T)` = 造一个 T 的零值，返回它的地址：

```go
p := new(int)

fmt.Println(*p)

*p = 10
fmt.Println(*p)
```

```text
0
10
```

日常其实不太用 `new`，更顺手的是字面量加取地址：

```go
u := User{Name: "alice"}
p := &u
// 或者一步到位：p := &User{Name: "alice"}
```

---

## 5. 传指针修改调用方变量

第 01 篇（数组指针）、第 07 篇（指针接收者）都用过这招了，收拢成一句话：

```go
func reset(n *int) {
    *n = 0
}

func main() {
    x := 10
    reset(&x)
    fmt.Println(x)
}
```

```text
0
```

严谨地说：Go 的传参**永远是值传递**，这里被复制的是"地址"这个值。地址的副本仍然指向同一个 `x`，所以函数能顺着它改到原变量。整个 Go 的传参规则至此可以统一成一句：**一律复制，区别只在复制的是数据本身，还是找到数据的线索（指针/slice 头/map 句柄）。**

---

## 6. 指向 struct

```go
type User struct {
    Name string
    Age  int
}

u := User{Name: "alice", Age: 18}
p := &u

p.Age++

fmt.Println(u.Age)
```

```text
19
```

`p.Age` 是简化写法，等价于 `(*p).Age`——第 07 篇讲过的自动解引用，对普通指针同样生效。所以 Go 代码里几乎看不到 `(*p).field` 这种写法。

## 7. 返回局部变量的地址：栈、堆与逃逸分析

带着 C/C++ 印象的人看到这段会心跳加速——返回局部变量的地址？

```go
func NewUser(name string) *User {
    u := User{Name: name}
    return &u              // 函数都结束了，这地址还能用？
}
```

在 Go 里**完全合法且安全**。原因：变量放栈还是堆，不由你写的类型决定，由编译器的**逃逸分析**（escape analysis）决定——它检查每个变量的地址会不会"逃出"函数，逃出去的就安排到堆上，由 GC 管理。

### 🕳️ 坑：值类型 = 栈，引用类型 = 堆？

以为会怎样：struct 是值类型所以在栈上，slice/map 是引用类型所以在堆上，用了指针就上堆。
实际怎样：**值/引用说的是复制语义（前几篇的主题），栈/堆是分配位置，两者互不决定。** 值类型的 struct 可以住堆，取了地址的变量也可以整辈子待在栈上。
为什么：分配位置只看一件事——这个变量的生命周期需不需要超出函数。

空口无凭，让编译器自己作证。`-gcflags` 可以把逃逸分析的判断打印出来（`-m` 显示分析结果，`-l` 关掉内联让输出更干净）：

```go
package main

import "fmt"

type User struct {
	Name string
}

func newUser(name string) *User {
	u := User{Name: name}
	return &u
}

func double(n int) int {
	p := &n     // 也取了地址
	*p = *p * 2 // 但地址没离开函数
	return n
}

func main() {
	u := newUser("alice")
	fmt.Println(u.Name)
	fmt.Println(double(10))
}
```

```bash
go build "-gcflags=-m -l" main.go
```

```text
# command-line-arguments
./main.go:9:14: leaking param: name
./main.go:10:2: moved to heap: u
./main.go:22:13: ... argument does not escape
./main.go:22:15: u.Name escapes to heap
./main.go:23:13: ... argument does not escape
./main.go:23:20: double(10) escapes to heap
```

只看两条关键证据：

- `./main.go:10:2: moved to heap: u`——`newUser` 里的 `u` 是个如假包换的**值类型 struct**，但它的地址被 return 带出了函数，所以被搬到堆上。**值类型 ≠ 栈。**
- `double` 里的 `p := &n` **没有任何一行 moved to heap**——取了地址，但地址没离开函数，`n` 和 `p` 全程住栈，函数结束原地回收，GC 都不用惊动。**用指针 ≠ 堆。**

（22、23 行那几条是 `fmt.Println` 自己的参数装箱信息，与本实验无关，忽略。）程序本身照常运行：

```text
alice
20
```

**一句话总结：值/引用回答"复制了什么"，栈/堆回答"住在哪"；前者你说了算（选类型），后者编译器说了算（逃逸分析）——你不需要、也没有手段手动管理。** 所以 Go 里"构造函数返回 `*User`"是标准写法，放心用。

## 8. 指针不能运算

C 程序员的 `p++` 在 Go 里过不了编译：

```go
package main

import "fmt"

func main() {
	x := 10
	p := &x
	p++
	fmt.Println(p)
}
```

```text
# command-line-arguments
./main.go:8:2: invalid operation: p++ (non-numeric type *int)
```

没有指针算术，就没有"算错偏移量捅进别人内存"这一整类灾难。需要遍历连续数据？那是数组、slice 和下标的工作。

---

## 9. 什么时候使用指针

适合用指针：

- 函数/方法需要修改调用方传入的值（第 5 节、第 07 篇的指针接收者）。
- struct 很大，频繁复制不划算。
- 表达"可能没有"：`*time.Time` 为 nil 表示没有时间——类似 JS 用 `null` 占位。
- 类型里有锁、文件、连接等不该被复制的资源。

不必用指针：

- 小的值（坐标、金额、配置快照）——复制比共享更省心。
- 单纯觉得"指针 = 高效"——第 7 节刚证明过，指针可能反而把变量推上堆，增加 GC 负担。
- 会让"这份数据现在归谁改"变得说不清的场合。

## 10. 指针、slice、map 的区别

模块学到这里，把"函数里能不能改到调用方数据"做个总对照实验：

```go
func changeSlice(s []int) {
    s[0] = 100           // 改共享底层数组里的元素
}

func grow(s []int) {
    s = append(s, 4)     // 只改了参数这个 slice 头副本
}

func changeMap(m map[string]int) {
    m["x"] = 100         // 改共享哈希表的内容
}

func main() {
    s := []int{1, 2, 3}
    changeSlice(s)
    fmt.Println(s)

    grow(s)
    fmt.Println(s, len(s))

    m := map[string]int{}
    changeMap(m)
    fmt.Println(m)
}
```

```text
[100 2 3]
[100 2 3] 3
map[x:100]
```

三个信息：改元素生效了、`grow` 的 append **没**生效（长度还是 3——append 结果只写进了参数副本的 slice 头，这就是第 03 篇"append 必须接返回值"的深层原因）、改 map 生效了。

汇总本模块的"复制了什么"全景表：

| 传入类型 | 复制的是 | 函数内改内容 | 函数内整个换掉/append |
|------|------|------|------|
| `int`、数组、struct | 全部数据 | 调用方看不见 | 看不见 |
| `*T` 指针 | 地址 | **看得见** | 换指向看不见 |
| `[]T` slice | slice 头 | 改元素**看得见** | append/重切看不见 |
| `map[K]V` | 句柄 | 增删改 key **看得见** | 换新表看不见 |

**Go 没有引用传递，只有"复制的东西里含不含线索"的区别。** 这张表就是整个模块内存语义的浓缩。

## 本篇重点

- [ ] `&` 取地址、`*` 解引用；`*int` 是类型、`*p` 是操作；nil 指针解引用 = panic。
- [ ] Go 传参一律值传递：传指针复制的是地址，顺着地址能改到原变量。
- [ ] 值/引用（复制语义）和栈/堆（分配位置）是两回事：值类型 struct 会因地址逃逸上堆，取了地址的变量也能全程住栈——逃逸分析说了算，`go build -gcflags="-m -l"` 能看到判决。
- [ ] 返回局部变量地址在 Go 完全安全，是构造函数的标准写法；指针不能做算术。
- [ ] 总对照表：int/数组/struct 复制数据，指针复制地址，slice 复制头，map 复制句柄——"含线索"的才能改到调用方数据。
