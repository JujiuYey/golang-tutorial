# CI 流水线与质量门禁

CI 的作用是把本地验证搬到统一环境里执行。它应该稳定、可读、失败信息明确。

---

## 1. CI 应该跑什么

基础门禁：

```bash
go mod download
go mod verify
go fmt ./...
go vet ./...
go test ./...
go test -race ./...
go build ./...
```

实际项目可以根据耗时拆分：

- 快速检查：fmt、vet、unit test。
- 深度检查：race、集成测试、镜像构建。

---

## 2. GitHub Actions 示例

```yaml
name: ci

on:
  push:
    branches: [main]
  pull_request:

jobs:
  verify:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Set up Go
        uses: actions/setup-go@v5
        with:
          go-version-file: go.mod
          cache: true

      - name: Download modules
        run: go mod download

      - name: Verify modules
        run: go mod verify

      - name: Format check
        run: |
          go fmt ./...
          git diff --exit-code

      - name: Vet
        run: go vet ./...

      - name: Test
        run: go test ./...

      - name: Race
        run: go test -race ./...

      - name: Build
        run: go build ./...
```

`go-version-file: go.mod` 可以让 CI 使用项目声明的 Go 版本。

---

## 3. 为什么格式检查要看 diff

CI 中运行：

```bash
go fmt ./...
git diff --exit-code
```

如果 `go fmt` 修改了文件，`git diff --exit-code` 会失败。这样可以提醒开发者在本地提交格式化后的代码。

---

## 4. 使用 Makefile 减少重复

CI 可以直接调用 Makefile：

```yaml
- name: Check
  run: make check
```

好处是本地和 CI 使用同一套命令。缺点是 CI 输出可能变粗，需要在 Makefile 中保持 target 清楚。

---

## 5. 分离耗时任务

如果 `-race` 很慢，可以拆成单独 job：

```yaml
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-go@v5
        with:
          go-version-file: go.mod
          cache: true
      - run: go test ./...

  race:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-go@v5
        with:
          go-version-file: go.mod
          cache: true
      - run: go test -race ./...
```

拆分后能更快看到是普通测试失败，还是 race 失败。

---

## 6. 上传覆盖率产物

```yaml
- name: Coverage
  run: go test -coverprofile=coverage.out ./...

- name: Upload coverage
  uses: actions/upload-artifact@v4
  with:
    name: coverage
    path: coverage.out
```

覆盖率文件可以作为排查测试盲区的证据。

---

## 7. CI 失败如何处理

CI 失败时按顺序看：

1. 哪个 job 失败。
2. 哪个 step 失败。
3. 失败命令是什么。
4. 本地能否复现同一命令。
5. 是代码问题、测试问题、环境问题，还是依赖问题。

不要只看最后一行。很多有价值的信息在失败前几十行。

---

## 8. 学习检查

检查 CI 时，重点看：

1. 是否使用 `go-version-file`。
2. 是否启用 module cache。
3. 是否检查 `go fmt` 后的 diff。
4. 是否运行 `go vet`。
5. 是否运行测试和 race。
6. 是否构建所有包或主程序。
7. CI 命令是否能在本地复现。
