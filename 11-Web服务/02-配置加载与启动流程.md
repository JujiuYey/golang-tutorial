# 02: 配置加载与启动流程

监听地址、超时、日志级别、数据库地址……这些值散落在代码里,换个环境就要改代码重新编译。这篇解决两个问题:**把配置集中到一个结构体、在启动那一刻就把配置错误暴露出来**,以及把 `http.Server` 的启动-关闭流程写对。学完你的服务能做到:配错了立刻报错退出,收到 Ctrl+C 或容器的 SIGTERM 能干净地停下来。

对比 Node:你熟悉的 `dotenv` + `process.env.PORT || 8080` + `app.listen(port)` 三件套,在 Go 里对应 `os.Getenv` + 配置结构体 + 显式的 `http.Server`。最大的差别是最后一个——Express 的 `app.listen` 背后什么超时都没配,Go 的习惯是**把超时和关闭流程明明白白写出来**。

---

## 1. 配置结构体

先定义服务真正需要的配置:

```go
package config

import "time"

type Config struct {
    Addr              string
    LogLevel          string
    ReadHeaderTimeout time.Duration
    WriteTimeout      time.Duration
    ShutdownTimeout   time.Duration
    DatabaseURL       string
}
```

注意字段类型:超时用 `time.Duration` 而不是 `int`——"10 是 10 秒还是 10 毫秒"这种事故,类型系统直接帮你消灭。

配置结构体表达的是**服务的需求**,不是环境变量的复刻。环境变量只是来源之一(以后还可能来自文件、命令行参数)。

---

## 2. 从环境变量加载

```go
func Load() (Config, error) {
    cfg := Config{
        Addr:              getenv("ADDR", ":8080"),
        LogLevel:          getenv("LOG_LEVEL", "info"),
        ReadHeaderTimeout: 5 * time.Second,
        WriteTimeout:      10 * time.Second,
        ShutdownTimeout:   10 * time.Second,
        DatabaseURL:       os.Getenv("DATABASE_URL"),
    }

    if raw := os.Getenv("READ_HEADER_TIMEOUT"); raw != "" {
        value, err := time.ParseDuration(raw)
        if err != nil {
            return Config{}, fmt.Errorf("parse READ_HEADER_TIMEOUT: %w", err)
        }
        cfg.ReadHeaderTimeout = value
    }
    // WRITE_TIMEOUT、SHUTDOWN_TIMEOUT 同样处理……

    if err := cfg.Validate(); err != nil {
        return Config{}, err
    }
    return cfg, nil
}

func getenv(key, fallback string) string {
    value := os.Getenv(key)
    if value == "" {
        return fallback
    }
    return value
}
```

三个细节:

- `os.Getenv` 拿到的**永远是 string**(和 `process.env` 一样),数字、时长都要自己解析。
- `time.ParseDuration` 认识 `500ms`、`5s`、`2m` 这类写法,比让用户填毫秒数友好且不易错。
- 解析失败立刻返回带上下文的错误(`%w` 包装,模块 03 的老朋友),而不是悄悄用默认值。

配错时真实的效果——`READ_HEADER_TIMEOUT=abc go run ./cmd/task-api`:

```text
2026/07/16 10:13:16 parse READ_HEADER_TIMEOUT: time: invalid duration "abc"
```

进程退出码 1,一行日志直指哪个变量、什么值、为什么不行。

## 3. 配置校验

加载完马上校验,把问题拦在启动阶段:

```go
func (c Config) Validate() error {
    if c.Addr == "" {
        return fmt.Errorf("ADDR is required")
    }
    if c.ReadHeaderTimeout <= 0 {
        return fmt.Errorf("READ_HEADER_TIMEOUT must be positive")
    }
    if c.WriteTimeout <= 0 {
        return fmt.Errorf("WRITE_TIMEOUT must be positive")
    }
    if c.ShutdownTimeout <= 0 {
        return fmt.Errorf("SHUTDOWN_TIMEOUT must be positive")
    }
    return nil
}
```

**一句话总结:配置错误要死在启动时,而不是活到半夜第一个请求进来时。**

## 4. 启动 http.Server

### 🕳️ 坑:直接 `http.ListenAndServe(":8080", handler)`

以为会怎样:一行起服务,和 `app.listen(8080)` 一样省事。
实际怎样:这个服务**没有任何超时**——一个客户端慢慢悠悠发请求头,连接就一直占着;也没有任何优雅关闭的入口。
为什么:包级函数 `http.ListenAndServe` 内部 new 了一个零值 `http.Server`,所有超时字段都是 0(不限制)。生产服务要自己构造:

```go
func (a *App) server() *http.Server {
    return &http.Server{
        Addr:              a.config.Addr,
        Handler:           a.Handler(),
        ReadHeaderTimeout: a.config.ReadHeaderTimeout,
        WriteTimeout:      a.config.WriteTimeout,
    }
}
```

监听地址、handler、超时都从 `App`(第 03 篇)身上拿,启动逻辑只要一个配置好的 `http.Server`。

---

## 5. 处理启动错误和关闭信号

完整的 `Run` 长这样,逐段拆:

```go
func (a *App) Run() error {
    ctx, stop := signal.NotifyContext(context.Background(), os.Interrupt, syscall.SIGTERM)
    defer stop()
    //  ────────┬────────
    //  ①收到 Ctrl+C 或 SIGTERM 时,这个 ctx 会被取消(模块 06 的 context 用在了进程级)

    server := a.server()
    errCh := make(chan error, 1)

    go func() {                                  // ②监听是阻塞的,放进 goroutine
        a.logger.Info("server listening", slog.String("addr", server.Addr))
        err := server.ListenAndServe()
        if err != nil && !errors.Is(err, http.ErrServerClosed) {
            errCh <- err                         // ③真正的启动失败(如端口被占)
            return
        }
        errCh <- nil
    }()

    select {                                     // ④两件事赛跑:启动失败 vs 收到信号
    case err := <-errCh:
        return err
    case <-ctx.Done():
        a.logger.Info("shutdown signal received")
        shutdownCtx, cancel := context.WithTimeout(context.Background(), a.config.ShutdownTimeout)
        defer cancel()

        if err := server.Shutdown(shutdownCtx); err != nil {
            return fmt.Errorf("shutdown server: %w", err)
        }
        a.logger.Info("server stopped")
        return nil
    }
}
```

对比 Node:`process.on('SIGTERM', () => server.close(...))` 是同一件事,但 Go 版本用 channel + select 把"启动失败"和"要求关闭"两条路径收敛到一个出口,不会出现回调里没人接的错误。

### 🕳️ 坑:把 `http.ErrServerClosed` 当成错误

以为会怎样:`ListenAndServe` 返回了非 nil 的 error,肯定出事了。
实际怎样:调用 `Shutdown` 之后,`ListenAndServe` **一定**返回 `http.ErrServerClosed`——这是"正常关闭"的信号,不是故障。
为什么:监听循环总得用一个返回值表达"我停了"。所以判断错误时必须排除它:`err != nil && !errors.Is(err, http.ErrServerClosed)`。漏掉这个判断,每次正常关停都会打一条假错误日志。

实跑效果(`ADDR=:19081` 启动,请求一次,再发 SIGTERM):

```text
time=2026-07-16T10:21:33.118-07:00 level=INFO msg="server listening" addr=:19081
time=2026-07-16T10:21:33.908-07:00 level=INFO msg="http request" method=GET path=/healthz status=200 bytes=16 duration=15.375µs request_id=e7d67ac80bc40c83710dd42f81d88307
time=2026-07-16T10:21:33.929-07:00 level=INFO msg="shutdown signal received"
time=2026-07-16T10:21:33.929-07:00 level=INFO msg="server stopped"
```

没有报错、没有 panic,四行日志讲完一个服务的一生。`Shutdown` 的更多细节(它到底等什么、等多久)留到第 09 篇。

## 6. main 中处理最终错误

`main` 没有返回值,错误没地方去,所以常见套路是把逻辑掏空到 `run`:

```go
func main() {
    if err := run(); err != nil {
        log.Fatal(err)
    }
}

func run() error {
    cfg, err := config.Load()
    if err != nil {
        return err
    }
    application, err := buildApp(cfg)
    if err != nil {
        return err
    }
    return application.Run()
}
```

好处有两个:`run` 是普通函数,可以被测试;错误能用 `%w` 逐层包装着流回来,最后由 `main` 统一打印退出。第 2 节那行 `parse READ_HEADER_TIMEOUT: ...` 的输出就是这么流出来的。

**一句话总结:main 只负责"err 不为 nil 就 Fatal",其余全在 run 里。**

---

## 本篇重点

- [ ] 配置集中进结构体,超时用 `time.Duration`;环境变量都是 string,解析失败要报错而不是吞掉。
- [ ] 配置加载后立刻 `Validate`,让错误死在启动时。
- [ ] 别用裸的 `http.ListenAndServe`——零值 `http.Server` 没有任何超时;显式构造并至少配 `ReadHeaderTimeout`。
- [ ] `signal.NotifyContext` 监听 SIGINT/SIGTERM,收到后调 `Shutdown`(带超时);`ListenAndServe` 返回的 `http.ErrServerClosed` 是正常关闭信号,不是错误。
- [ ] `main` 掏空成 `run() error`,错误统一从一个出口离开。

## 练习

给你的 `book-api` 写配置和启动流程:

要求:

1. `config.Load` 支持 `ADDR`(默认 `:8080`)、`LOG_LEVEL`(默认 `info`)、`SHUTDOWN_TIMEOUT`(默认 `10s`,支持 `time.ParseDuration` 格式)。
2. `Validate` 拒绝空 `ADDR` 和非正的超时。
3. 显式创建 `http.Server`,配置 `ReadHeaderTimeout`,先注册一个 `GET /healthz` 路由。
4. 支持 `Ctrl+C` 优雅退出,退出前后各打一条日志。
5. 验证三件事:`SHUTDOWN_TIMEOUT=xyz` 启动会报错退出;正常启动后 `curl` 能通;`Ctrl+C` 后看到两条关闭日志。

提示:goroutine 里跑 `ListenAndServe`,主流程 `select` 等 `errCh` 和 `ctx.Done()`(第 5 节的骨架);别忘了排除 `http.ErrServerClosed`。
