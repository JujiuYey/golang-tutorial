# 07: 中间件链:日志、恢复与认证入口

请求日志、panic 恢复、请求 ID、认证、CORS……这些逻辑不属于任何一个业务 handler,却要罩住所有路由。这篇解决的问题是:**用标准库把中间件写成可组合的链**,并搭好三件必备中间件:请求 ID、访问日志、panic 恢复。

对比 Node:Express 的 `app.use(logger)`、Koa 的洋葱模型你都熟。Go 标准库没有 `app.use`,但中间件的本质两边完全一样——**包一层的函数**。Koa 的 `await next()` 前后各写一段,对应 Go 里 `next.ServeHTTP(w, r)` 前后各写一段。

---

## 1. 中间件类型

标准库世界里,中间件就是一个函数:收下一个 handler,吐出一个新 handler:

```go
package httpx

import "net/http"

type Middleware func(http.Handler) http.Handler
```

拆开看:

```text
type Middleware func(http.Handler) http.Handler
//                   ─────┬──────  ──────┬─────
//                   ①下一棒(next)   ②包了一层的新 handler
```

Express 的 `(req, res, next) => {...}` 是运行时靠 `next()` 交棒;Go 是**构造时**就把 handler 一层层套好,请求来了顺着套娃往里钻。

---

## 2. 组合中间件

手动套三层就没法看了:`m1(m2(m3(handler)))`。写一个 `Chain` 抹平:

```go
func Chain(handler http.Handler, middlewares ...Middleware) http.Handler {
    for i := len(middlewares) - 1; i >= 0; i-- {
        handler = middlewares[i](handler)
    }
    return handler
}
```

倒着循环是为了让**参数顺序 = 请求进入顺序**。写个带打印的中间件实跑验证:

```go
func tag(name string) httpx.Middleware {
    return func(next http.Handler) http.Handler {
        return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
            fmt.Println("进入", name)
            next.ServeHTTP(w, r)
            fmt.Println("离开", name)
        })
    }
}

handler := httpx.Chain(app, tag("A"), tag("B"), tag("C"))
```

```text
进入 A
进入 B
进入 C
--- handler 处理请求 ---
离开 C
离开 B
离开 A
```

进入 A→B→C,离开 C→B→A——这就是 Koa 洋葱模型,Go 版不需要 async/await,函数调用栈天然就是洋葱。

实际服务的链:

```go
handler := httpx.Chain(
    a.mux,
    httpx.RequestID(),      // 最先:后面所有人都要用 request ID
    httpx.Recover(logger),  // 其次:罩住后面所有 panic
    httpx.AccessLog(logger),
)
```

**一句话总结:Chain 参数从左到右 = 请求进入顺序;谁的产出被更多人依赖,谁排前面。**

## 3. 请求 ID 中间件

请求 ID 用来把一个请求产生的所有日志串成一条线:

```go
type requestIDKey struct{}

func RequestID() Middleware {
    return func(next http.Handler) http.Handler {
        return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
            id := r.Header.Get("X-Request-ID")
            if id == "" {                       // 客户端没带就生成一个
                generated, err := newRequestID()
                if err != nil {
                    WriteErrorWithCode(w, http.StatusInternalServerError, "internal_error", "failed to create request id")
                    return
                }
                id = generated
            }

            w.Header().Set("X-Request-ID", id)  // 写回响应头,方便客户端对账
            ctx := context.WithValue(r.Context(), requestIDKey{}, id)
            next.ServeHTTP(w, r.WithContext(ctx))
        })
    }
}

func RequestIDFromContext(ctx context.Context) string {
    value, ok := ctx.Value(requestIDKey{}).(string)
    if !ok {
        return ""
    }
    return value
}
```

实测透传(客户端带 ID 进来,原样带回去):

```bash
curl -si -H 'X-Request-ID: req-123' http://localhost:19080/healthz | grep -i x-request-id
```

```text
X-Request-Id: req-123
```

两个细节:

- context key 用**未导出的空结构体类型** `requestIDKey{}`,别的包造不出同类型的 key,天然防冲突——这是 `context.WithValue` 的标准姿势(模块 06 第 07 篇)。
- 对比 JS:往 context 塞值 = Express 往 `res.locals` / `req` 上挂属性,但 Go 的版本是类型安全的,取出来要走 `RequestIDFromContext` 这道有类型断言的门。

## 4. 捕获状态码的 ResponseWriter

访问日志要记状态码,可 `http.ResponseWriter` 是只写的——写完你也不知道写了什么。办法:结构体嵌入(模块 05)包一层,顺手记下来:

```go
type statusRecorder struct {
    http.ResponseWriter        // 嵌入:没覆写的方法直接透传
    status int
    bytes  int
}

func (r *statusRecorder) WriteHeader(status int) {
    if r.status != 0 {
        return                 // 只认第一次
    }
    r.status = status
    r.ResponseWriter.WriteHeader(status)
}

func (r *statusRecorder) Write(data []byte) (int, error) {
    if r.status == 0 {
        r.status = http.StatusOK   // 没显式 WriteHeader 就 Write,隐式 200
    }
    n, err := r.ResponseWriter.Write(data)
    r.bytes += n
    return n, err
}
```

🕳️ 坑:handler 不调 `WriteHeader` 直接 `Write` 是合法的(隐式 200),所以记录器要把这种情况也当成 200,否则日志里会出现"状态码 0"。

## 5. 访问日志中间件

```go
func AccessLog(logger *slog.Logger) Middleware {
    return func(next http.Handler) http.Handler {
        return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
            start := time.Now()
            recorder := &statusRecorder{ResponseWriter: w}

            next.ServeHTTP(recorder, r)     // 洋葱:先放行,回来再记

            status := recorder.status
            if status == 0 {
                status = http.StatusOK
            }

            logger.Info(
                "http request",
                slog.String("method", r.Method),
                slog.String("path", r.URL.Path),
                slog.Int("status", status),
                slog.Int("bytes", recorder.bytes),
                slog.Duration("duration", time.Since(start)),
                slog.String("request_id", RequestIDFromContext(r.Context())),
            )
        })
    }
}
```

真实日志长这样(morgan 的结构化版):

```text
time=2026-07-16T10:14:04.298-07:00 level=INFO msg="http request" method=POST path=/tasks status=201 bytes=141 duration=1.043375ms request_id=e2a0d4b3c5fe13d6896785ee8d4b86fc
```

字段名(`method`、`path`、`status`……)要**稳定**——日志检索、看板、告警规则全都挂在这些字段名上,改一次全断。

## 6. panic 恢复中间件

handler 里一个 nil map 写入就能 panic。没人接的话,这个 goroutine 崩掉、连接被粗暴掐断。Recover 中间件是最后防线(模块 03 的 recover 终于有了最正当的用武之地):

```go
func Recover(logger *slog.Logger) Middleware {
    return func(next http.Handler) http.Handler {
        return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
            defer func() {
                value := recover()
                if value == nil {
                    return
                }

                logger.Error(
                    "panic recovered",
                    slog.Any("panic", value),
                    slog.String("stack", string(debug.Stack())),
                    slog.String("method", r.Method),
                    slog.String("path", r.URL.Path),
                    slog.String("request_id", RequestIDFromContext(r.Context())),
                )
                WriteErrorWithCode(w, http.StatusInternalServerError, "internal_error", "internal server error")
            }()

            next.ServeHTTP(w, r)
        })
    }
}
```

拿一个故意 `panic("boom")` 的路由实测:

```bash
curl -si http://localhost:19080/panic | head -2
curl -s  http://localhost:19080/panic
```

```text
HTTP/1.1 500 Internal Server Error
Content-Type: application/json; charset=utf-8
{"error":{"code":"internal_error","message":"internal server error"}}
```

服务端日志(stack 很长,截取开头):

```text
time=2026-07-16T10:15:14.336-07:00 level=ERROR msg="panic recovered" panic=boom stack="goroutine 62 [running]:\nruntime/debug.Stack()..." method=GET path=/panic request_id=...
```

客户端拿到干净的 500,服务端留下完整堆栈,进程继续活着。🕳️ 提醒:Recover 是**兜底**,不是错误处理——预期内的失败仍然走上一篇的 error 链路,别用 panic 传业务错误。

## 7. 认证入口中间件

认证也是横切逻辑,同样做成中间件——核心是依赖一个**可替换的校验接口**:

```go
type TokenVerifier interface {
    Verify(ctx context.Context, token string) (Principal, error)
}

func Middleware(verifier TokenVerifier) func(http.Handler) http.Handler {
    return func(next http.Handler) http.Handler {
        return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
            header := r.Header.Get("Authorization")
            token, ok := strings.CutPrefix(header, "Bearer ")
            if !ok || token == "" {
                httpx.WriteErrorWithCode(w, http.StatusUnauthorized, "unauthorized", "missing bearer token")
                return    // 不放行:到此为止
            }

            principal, err := verifier.Verify(r.Context(), token)
            if err != nil {
                httpx.WriteErrorWithCode(w, http.StatusUnauthorized, "unauthorized", "invalid bearer token")
                return
            }

            next.ServeHTTP(w, r.WithContext(WithPrincipal(r.Context(), principal)))
        })
    }
}
```

套路和请求 ID 一模一样:解析凭证 → 校验 → 把身份(`Principal`)塞进 context → 放行;失败就写响应并 `return`(**不调 `next` 就是拦截**,对应 Express 里不调 `next()`)。它住在独立的 `auth` 包里;`Principal`、授权检查、token 签名这些细节第 10 篇展开。

实测拦截效果:

```bash
curl -si http://localhost:19080/admin/ping | head -1
curl -si -H 'Authorization: Bearer admin-token' http://localhost:19080/admin/ping | head -1
```

```text
HTTP/1.1 401 Unauthorized
HTTP/1.1 200 OK
```

## 本篇重点

- [ ] 中间件 = `func(http.Handler) http.Handler`,构造时套娃;`next.ServeHTTP` 前后各写一段 = Koa 洋葱的进入/离开。
- [ ] `Chain` 让参数顺序 = 请求进入顺序;RequestID 最先、Recover 其次,让 panic 日志里也有 request ID。
- [ ] context key 用未导出空结构体;往 context 塞值 = 挂 `res.locals`,但取出要过类型断言的门。
- [ ] 记录状态码要用嵌入 `ResponseWriter` 的 recorder,并处理"不调 WriteHeader 直接 Write = 隐式 200"。
- [ ] Recover 是 panic 兜底(客户端 500、日志留堆栈、进程活着),业务错误仍走 error 链路;中间件不放行就是拦截。

要求:

1. 实现 `Chain`、`RequestID`、`AccessLog`、`Recover` 并接到 `App.Handler()` 上。
2. 加一个故意 panic 的路由,curl 验证:响应是 500 的统一 JSON、日志有堆栈和 request_id、服务还能继续处理下一个请求。
3. 新写一个 `Timing` 中间件:响应头里加 `X-Response-Time`(耗时毫秒数)。想清楚它该排在链的哪个位置。
4. 用带打印的 tag 中间件(第 2 节)验证你的链的进入/离开顺序符合预期。

提示:`X-Response-Time` 必须在 handler 写响应**之前**设置头(第 05 篇 5 节的顺序坑)——所以耗时只能测到"进入 handler 前",或者你需要 statusRecorder 那样的包装;两种取舍都试试。
