# 06: 统一错误处理与状态码

同一种"任务不存在",A handler 回 404、B handler 回 500、C handler 回 200 带个 error 字段——这是 Web 服务最容易烂掉的地方。这篇解决的问题是:**建一个统一的错误模型,让"业务错误 → HTTP 状态码"的映射只写在一个地方**。

对比 Express:你习惯 `next(err)` 把错误甩给最后那个 `(err, req, res, next)` 错误中间件集中处理。Go 的标准库 handler 没有 `next`,也不返回 error——但我们可以自己造出同样的"集中处理"体验,这就是第 4 节的 `Wrap`。

---

## 1. 业务错误和 HTTP 错误分开

service 层定义业务错误(模块 03 的哨兵错误模式):

```go
package task

import "errors"

var (
    ErrNotFound      = errors.New("task not found")
    ErrAlreadyDone   = errors.New("task already done")
    ErrInvalidStatus = errors.New("invalid task status")
)
```

注意这些错误里**没有** 404、409 这样的数字。service 不知道也不该知道 HTTP 的存在——同一个 `ErrNotFound`,HTTP 入口翻译成 404,CLI 入口翻译成退出码,将来的 gRPC 入口翻译成 `codes.NotFound`。

**一句话总结:service 说"发生了什么",HTTP 层决定"用什么状态码说"。**

---

## 2. 在 HTTP 边界映射状态码

映射集中在一个函数里,handler 出错就调它:

```go
func writeTaskError(w http.ResponseWriter, err error) {
    switch {
    case errors.Is(err, ErrNotFound):
        httpx.WriteErrorWithCode(w, http.StatusNotFound, "task_not_found", "task not found")
    case errors.Is(err, ErrAlreadyDone):
        httpx.WriteErrorWithCode(w, http.StatusConflict, "task_already_done", "task already done")
    case errors.Is(err, ErrInvalidStatus):
        httpx.WriteErrorWithCode(w, http.StatusUnprocessableEntity, "invalid_status", "invalid task status")
    default:
        httpx.WriteErrorWithCode(w, http.StatusInternalServerError, "internal_error", "internal server error")
    }
}
```

实测两条路径:

```bash
curl -si http://localhost:19080/tasks/nope | head -1
curl -s  http://localhost:19080/tasks/nope
curl -si -X POST http://localhost:19080/tasks/task_1/complete | head -1   # task_1 已是 done
curl -s  -X POST http://localhost:19080/tasks/task_1/complete
```

```text
HTTP/1.1 404 Not Found
{"error":{"code":"task_not_found","message":"task not found"}}
HTTP/1.1 409 Conflict
{"error":{"code":"task_already_done","message":"task already done"}}
```

用 `errors.Is` 而不是 `==`,是因为错误可能在向上传的路上被 `%w` 包了几层(第 6 节)——`Is` 能穿透包装找到根因。

🕳️ 坑:注意 `default` 分支。**没被认出来的错误一律 500 + 通用文案**,绝不把 `err.Error()` 直接吐给客户端——里面可能有 SQL 语句、文件路径这类内部细节。

---

## 3. 定义 AppError

错误种类多了,switch 越写越长,可以升级成携带元信息的统一错误类型(模块 03 的自定义错误类型 + `Unwrap`):

```go
type AppError struct {
    Status  int
    Code    string
    Message string
    Err     error
}

func (e *AppError) Error() string {
    if e.Err == nil {
        return e.Message
    }
    return fmt.Sprintf("%s: %v", e.Message, e.Err)
}

func (e *AppError) Unwrap() error {
    return e.Err
}
```

使用时把底层错误包进去:

```go
return &httpx.AppError{
    Status:  http.StatusBadRequest,
    Code:    "invalid_json",
    Message: "invalid json body",
    Err:     err,
}
```

分工:对外只返回 `Message`,日志记录完整的 `Err`。实跑验证它和 `errors` 家族的配合:

```go
inner := errors.New(`decode json: unknown field "hack"`)
err := &httpx.AppError{Status: 400, Code: "invalid_json", Message: "invalid json body", Err: inner}

fmt.Println("Error():", err.Error())

var appErr *httpx.AppError
wrapped := fmt.Errorf("create handler: %w", error(err))
fmt.Println("errors.As 找回:", errors.As(wrapped, &appErr), appErr.Status, appErr.Code)
fmt.Println("errors.Is 找到根因:", errors.Is(wrapped, inner))
```

```text
Error(): invalid json body: decode json: unknown field "hack"
errors.As 找回: 400 invalid_json
errors.Is 找到根因: true
```

再包一层也不怕:`errors.As` 能从包装链里把 `*AppError` 捞出来,`errors.Is` 能一路钻到根因——`Unwrap` 方法就是为它们修的路。

## 4. 让 handler 返回 error

标准库 handler 签名没有返回值,错误处理只能"写响应 + return"两行两行地写。包一层,把 Express 错误中间件的体验搬过来:

```go
type HandlerFunc func(http.ResponseWriter, *http.Request) error
//                                                       ──┬──
//                                    ①自定义签名:允许返回 error

func Wrap(logger *slog.Logger, fn HandlerFunc) http.HandlerFunc {
    return func(w http.ResponseWriter, r *http.Request) {
        if err := fn(w, r); err != nil {
            WriteMappedError(logger, w, r, err)   // ②错误集中到一个出口
        }
    }
}

func WriteMappedError(logger *slog.Logger, w http.ResponseWriter, r *http.Request, err error) {
    var appErr *AppError
    if errors.As(err, &appErr) {                  // ③认识的错:按元信息返回
        WriteErrorWithCode(w, appErr.Status, appErr.Code, appErr.Message)
        return
    }

    logger.Error(                                 // ④不认识的错:记日志 + 500
        "unhandled request error",
        slog.String("method", r.Method),
        slog.String("path", r.URL.Path),
        slog.String("error", err.Error()),
    )
    WriteErrorWithCode(w, http.StatusInternalServerError, "internal_error", "internal server error")
}
```

业务 handler 从此可以线性地写,`return` 就是抛错:

```go
func (h *HTTPHandler) Create(w http.ResponseWriter, r *http.Request) error {
    var req createTaskRequest
    if err := httpx.DecodeJSON(w, r, &req, 1<<20); err != nil {
        return &httpx.AppError{
            Status: http.StatusBadRequest, Code: "invalid_json",
            Message: "invalid json body", Err: err,
        }
    }

    if err := req.Validate(); err != nil {
        return &httpx.AppError{
            Status: http.StatusUnprocessableEntity, Code: "invalid_request",
            Message: err.Error(), Err: err,
        }
    }

    task, err := h.service.Create(r.Context(), req.ToInput())
    if err != nil {
        return mapTaskError(err)
    }

    return httpx.WriteJSON(w, http.StatusCreated, taskResponseFromModel(task))
}
```

注册时套一下:`mux.HandleFunc("POST /tasks", httpx.Wrap(logger, h.Create))`。

对比 JS:`Wrap` = Express 的错误中间件 + `express-async-handler` 二合一;`return &AppError{...}` = `throw new HttpError(400, ...)`。区别是 Go 里这条链路完全是你自己的 40 行代码,没有框架魔法。

## 5. 错误码要稳定

错误响应里带机器可读的 `code`:

```json
{
  "error": {
    "code": "task_not_found",
    "message": "task not found"
  }
}
```

分工:`message` 给人看,可以随时改得更友好;`code` 给程序看,前端 `if (err.code === 'task_not_found')`、SDK、测试断言都挂在它上面——**一旦发布就不要改**。别让客户端去解析 message 字符串,那等于把你的错误文案变成了 API 契约。

---

## 6. 错误日志不要重复打

### 🕳️ 坑:每一层都 log 一次

以为会怎样:多打点日志总没坏处。
实际怎样:一个数据库错误在 repository、service、handler 各打一条,同一件事故三条日志,查起来反而不知道哪条是源头,告警也翻三倍。
为什么:日志的职责应该归最外层,中间层的职责是**把上下文包进错误里带上去**:

- repository / service:返回错误(用 `%w` 加上下文),不打日志。
- HTTP 最外层(`WriteMappedError` 或 Recover 中间件):统一记录未处理错误。
- 预期内的业务错误(404 之类)不值得 error 级日志。

包装上下文的写法:

```go
task, err := repo.FindByID(ctx, id)
if err != nil {
    return Task{}, fmt.Errorf("find task by id %q: %w", id, err)
}
```

最外层拿到的错误长这样:`find task by id "abc": task not found`——上下文全在错误链里,`errors.Is(err, ErrNotFound)` 照样能认出根因(第 2 节的映射不受影响)。

**一句话总结:错误一路带着上下文向上流,日志只在出口打一次。**

---

## 本篇重点

- [ ] service 定义协议无关的哨兵错误;"错误 → 状态码"的映射只写在 HTTP 边界一处。
- [ ] 映射用 `errors.Is`(穿透 `%w` 包装),不认识的错误一律 500 + 通用文案,别泄露内部细节。
- [ ] `AppError` 携带 Status/Code/Message/Err,实现 `Unwrap` 后 `errors.As`、`errors.Is` 都能配合。
- [ ] 自定义 `func(w, r) error` 签名 + `Wrap`,handler 用 `return` 抛错 = Express 错误中间件的 Go 版。
- [ ] `code` 是机器契约,发布后不改;错误用 `%w` 层层带上下文,日志只在最外层打一次。
