# 教程文章风格指南

本仓库所有模块文章(`XX-模块名/NN-文章名.md`)的写作/改写标准。目标:**像一个人在教,而不是一份规格书**。读者是有 JS 经验、零 Go 基础的开发者,自学为主。

范例:`02-类型状态与控制流/05-字符串byte与rune.md` 第 6-7 节、`02-类型状态与控制流/07-布尔表达式与条件判断.md` 第 5 节、`03-函数错误与边界/02-多返回值与命名返回值.md` 第 4 节。

## 十条规则

### 1. 开篇一句"这篇解决什么问题"
文章开头 1-2 句话说清:学完这篇你能干什么/避开什么坑。不要以定义开头。

### 2. 大白话先行,术语跟后
先用生活化语言把机制说明白,再给出 Go 术语(括号或紧随其后)。
- ✅ "把字符串'倒进'两种不同的切片容器里(类型转换)"
- ❌ "类型转换是将一种类型的值转换为另一种类型的操作"

### 3. 所有示例代码必须带真实输出
每个代码块后面跟 ```text 输出块,内容必须是 `go run` 实际跑出来的(包括编译错误的原文)。**禁止凭记忆编造输出。**

### 4. 陌生语法逐段拆解
读者第一次见的语法,用 ASCII 标注图拆开:

```text
if ok := true; ok {
//  ─────┬────  ┬
//   ①先执行   ②再判断(必须是 bool)
```

### 5. 与 JS 对比
凡是和 JS 语义不同的地方(truthy/falsy、数组赋值、switch 贯穿、字符串索引…),用一行点出差异。JS 里有对应物的,用它做记忆锚点(如:if 初始化语句 = for 循环第一段)。

### 6. 坑要标出来
易错点用明确的小节或 "🕳️ 坑:" 前缀标注,写清"以为会怎样 → 实际怎样 → 为什么"。

### 7. 每节收口:一句话总结或对照表
概念对比(byte vs rune、值 vs 引用)用表格;单一机制用加粗的一句话总结。

### 8. 每篇末尾加"本篇重点"
3-5 条 checklist,读者扫一眼就知道该带走什么。放在练习之前。

### 9. 示例短、可运行、聚焦
单个示例 ≤ 20 行,`go run` 直接能跑。不贴生产级长代码。一个示例只证明一个点。

### 10. 保留结构与练习
- 保留原文的编号小节结构和知识点覆盖面(可合并/拆分小节,但不删知识点)。
- 保留每篇的练习题,不给出答案;可以补充提示。
- 语气可以直接称"你";允许适度口语,但不啰嗦。
