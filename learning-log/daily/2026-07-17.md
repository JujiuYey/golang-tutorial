## 主题：04-02 slice 基础（"slice 头"概念、第 6 节切片表达式）

### 起点
- 从 04-02 第 1 节 `sliceHeader struct` 定义开始,卡在两件事:
  1. **把 sliceHeader 当成接口**(觉得跟 `error` 一样)→ 其实是 struct,因为关键字是 `struct` 不是 `interface`。
  2. **用 JS 的"值类型 vs 引用类型"去套 Go**,而两套语言对数组的归类正好相反:
     - JS:数组 = 引用类型
     - Go:`[3]int` 数组 = 值类型,`[]int` slice = "值类型 + 含指针字段"

### 关键讲透的内容

#### slice 头的真实身份
- slice 头不是神秘的"内建运行时结构",本质就是**一个含指针字段的 struct**,跟下面这个 `Box` 机制完全一样:

  ```go
  type Box struct {
      name string
      ref  *int
  }
  ```
- slice 头就是 `Box` 把字段从 `name/ref` 换成 `data/len/cap`。值类型、复制独立、靠指针字段共享底层——这一套性质没多任何花活。
- "slice 是引用类型"的来源只有一个,就是 `data *T` 这一个指针字段。

#### 用 `d := a` + `d = append(d, 7, 8)` 跑两个时刻
- 时刻 1 (`d := a`): 两份独立 struct,data 字段值这一刻相同(都是 nil)
- 时刻 2 (`d = append`): cap=0 装不下 → 分配新数组 → d 的 data 字段被改 → a 的 data 字段从未被碰过 → a 仍是 nil
- 全程没有"对象身份"戏码:JS 的"两个变量指向同一对象"模型完全不适用

#### 打印类型 (`fmt.Printf("%T\n", x)`)
- `fmt.Println` 不解读格式化动词,`fmt.Println("%T", b)` 会原样打印 `%T [1 2 3]`
- 看类型必须用 `fmt.Printf("%T", ...)` 或 `fmt.Sprintf("%T", ...)`
- 速查卡:`Println(x)` 直接打 / `Printf("fmt", x)` 带动词 / `Sprintf` 返回字符串

### 易错点
- **slice 头是值类型 + 它内部含指针字段,这两个事实同时存在但要分开想**——只看一边都会得到错的直觉。
- `d := a` 时,以为是"赋值给同一对象"导致 a 跟着 d 变。错。a 是独立 struct,只是拷贝时三个字段值跟 d 一样。改 d 的 data 字段(append 搬家 / `d = d[:n]`)不会自动同步给 a。
- 用 `Println` 写格式化字符串,忘了 `Println` 不解析 `%T` 这类动词。

### 检查题结果
题来自我自己的出题(d := a / append / c[0] 修改三件套):
- (1)(2)(3) 类型:用户**已正确认出来**所有 `[]int` 都是同一个 slice 类型,包括 `arr[1:4]` 切出来的。
- (4) d 和 a 是"同一个 slice 头"吗:用户答"不是,独立的",直觉对,但表述需要分层说(独立 struct / 字段值是否相同)。

- (6) `c[0] = 999` 后 arr 变化:用户答会变,正确(c 的 data 指针指向 arr[1])。

用户自述 (4)-(6) "瞎蒙",实际靠直觉答对——接下来要刻意把"复制 / 共享 / 搬家"三种动作分类固化。
