# 04: 数字类型与运算

这篇解决的问题：JS 里所有数字都是一个 `number`，Go 里数字分了十几种——怎么选、边界在哪、运算时会踩什么坑（整数除法截断、溢出回绕、浮点误差）。

---

## 1. 整数类型

有符号整数（能表示负数）：

```go
var a int = 10
var b int8 = 127
var c int16 = 32767
var d int32 = 2147483647
var e int64 = 9223372036854775807
```

无符号整数（只能是 0 和正数，换来翻倍的正数范围）：

```go
var a uint = 10
var b uint8 = 255
var c uint16 = 65535
var d uint32 = 4294967295
var e uint64 = 18446744073709551615
```

类型名里的数字就是二进制位数，位数决定范围。`int` 和 `uint` 不带数字，大小依赖平台（64 位机器上就是 64 位）。

**选型口诀：写业务代码无脑用 `int`**；只有和外部世界对齐时（数据库字段、二进制协议、JSON 数值范围）才用明确位数的类型。

---

## 2. byte、rune、uintptr

两个别名，后面字符串篇（02-05）的主角：

```go
var b byte = 255  // byte 是 uint8 的别名，表示一个字节
var r rune = '你' // rune 是 int32 的别名，表示一个 Unicode 码点
```

`uintptr` 用于保存指针的整数表示，普通业务代码几乎用不到，认识即可。

---

## 3. 整数溢出

### 🕳️ 坑：超出范围不报错，而是"绕回去"

以为会怎样：`255 + 1 = 256`，或者报个错。
实际怎样：运行时溢出会静默回绕，255 加 1 变成 0。
为什么：`uint8` 只有 8 个二进制位，`11111111` 再进一位装不下，高位被丢掉，剩 `00000000`。

```go
func main() {
    var n uint8 = 255
    n++
    fmt.Println(n)
}
```

```text
0
```

想象汽车里程表：只有 8 个格子，打满之后再走一公里就归零重来。

不过，**编译期能看出来的溢出会直接拦下**——常量赋值超范围过不了编译：

```go
var n uint8 = 256
```

```text
编译错误：cannot use 256 (untyped int constant) as uint8 value in variable declaration (overflows)
```

对比 JS：JS 的 `number` 范围大到日常感知不到边界（超了也只是精度丢失）。Go 的整数边界是实打实的，**运行时溢出静默回绕、编译期溢出直接报错**。

---

## 4. 浮点数

Go 常用 `float64`——恰好就是 JS `number` 的底层格式（IEEE 754 双精度），所以 JS 的浮点坑 Go 全都有：

```go
func main() {
    price := 19.9
    fmt.Printf("%T\n", price)

    a, b := 0.1, 0.2
    fmt.Println(a + b)
}
```

```text
float64
0.30000000000000004
```

熟悉的 `0.30000000000000004`，和 JS 里一模一样：0.1 和 0.2 在二进制里都是无限循环小数，存不精确。

### 顺带一个冷知识：直接写 `0.1 + 0.2` 反而是 0.3

```go
fmt.Println(0.1 + 0.2) // 输出 0.3
```

因为 `0.1 + 0.2` 是两个**常量**相加，Go 编译器用任意精度在编译期算完（上一篇说的"常量是还没定型的数字"），最后才转成 float64。变量相加才会暴露浮点误差。

**实践规则：涉及金额，不要用浮点数存"元"，用整数存"分"**（最小货币单位），需要展示时再除。

---

## 5. 复数

Go 原生支持复数（JS 没有这东西）：

```go
func main() {
    z := 3 + 4i
    fmt.Println(real(z))
    fmt.Println(imag(z))
}
```

```text
3
4
```

科学计算领域用，日常业务代码几乎不碰，知道有即可。

---

## 6. 算术运算

`+ - * / %` 五件套。重点看除法：

```go
func main() {
    a, b := 10, 3

    fmt.Println(a + b)
    fmt.Println(a - b)
    fmt.Println(a * b)
    fmt.Println(a / b)
    fmt.Println(a % b)
}
```

```text
13
7
30
3
1
```

### 🕳️ 坑：整数除法直接丢掉小数

以为会怎样：`10 / 3 = 3.333...`（JS 里就是这样）。
实际怎样：得到 `3`，小数部分被直接截断（不是四舍五入）。
为什么：两个 `int` 相除，结果还是 `int`——Go 不会为了保住小数偷偷把类型升成浮点。

想要小数结果，先把操作数转成浮点：

```go
func main() {
    a, b := 10, 3
    fmt.Println(float64(a) / float64(b))
}
```

```text
3.3333333333333335
```

**一句话总结：Go 的 `/` 看操作数类型行事——int 除 int 得 int，想要小数先转 float64。**

---

## 7. 自增和自减

Go 有 `++` 和 `--`，但它们被降级成了**语句**，不能像 JS 那样塞进表达式里：

```go
func main() {
    n := 1
    n++
    n--
    fmt.Println(n)
}
```

```text
1
```

JS 里合法的 `x = n++`、`arr[i++]`，Go 直接拒绝：

```go
x := n++
```

```text
编译错误：syntax error: unexpected ++ at end of statement
```

顺带：Go 只有后置 `n++`，没有前置 `++n`。这个设计消灭了 JS 面试题里"前置后置返回值"那一整类困惑——`n++` 不返回任何值，只是让 n 加一，没有歧义。

---

## 8. 位运算

直接在二进制位上做运算，符号和 JS 完全一致（除了 JS 的 `>>>` Go 没有）：

```go
func main() {
    a, b := 5, 3 // 5=101, 3=011

    fmt.Println(a & b)  // 与：两边都是 1 才是 1
    fmt.Println(a | b)  // 或：任一边是 1 就是 1
    fmt.Println(a ^ b)  // 异或：不同才是 1
    fmt.Println(a << 1) // 左移一位 = 乘 2
    fmt.Println(a >> 1) // 右移一位 = 除 2（截断）
}
```

```text
1
7
6
10
2
```

拆一个看：

```text
  101  (5)
& 011  (3)
─────
  001  (1)   两列都是 1 的只有最后一列
```

位运算常见于权限标志（上一篇的 `1 << iota`）、二进制协议、性能敏感的底层代码。

---

## 本篇重点

- [ ] 业务代码整数无脑用 `int`；对齐外部格式（数据库、协议）时才用 `int64`/`uint8` 等定长类型。
- [ ] 运行时整数溢出静默回绕（255+1=0），编译期常量溢出直接报错——没有任何运行时警告。
- [ ] `float64` 就是 JS 的 `number`，浮点误差原样存在；金额用整数存最小单位（分）。
- [ ] int 除 int 结果还是 int，小数被截断；想要小数先 `float64()` 转换。
- [ ] `n++` 是语句不是表达式，不能 `x := n++`；也没有前置 `++n`。

## 练习

写一个百分比计算函数：

```go
func percent(part int, total int) float64
```

要求：

1. `total == 0` 时返回 `0`。
2. 正常情况下返回 `part / total * 100` 的浮点结果。
3. 测试 `percent(1, 3)`，确认没有被整数除法截断。

提示：想想 `float64(part) / float64(total)` 和 `float64(part / total)` 的区别——转换发生在除法之前还是之后，结果天差地别。
