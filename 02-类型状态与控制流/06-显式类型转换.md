# 06: 显式类型转换

这篇解决的问题：JS 里 `"1" + 2` 得 `"12"`、`if("0")` 走 true 分支——隐式转换的规则表你可能背过好几张。Go 的答案简单粗暴：**一张表都不用背，因为根本没有隐式转换**。任何跨类型操作都要你亲手写出来，这篇讲怎么写、哪里有坑。

转换语法只有一种：`目标类型(值)`，像函数调用一样。

---

## 1. 数字类型转换

连 `int` 和 `int64` 这种"亲兄弟"都不能直接混算：

```go
package main

import "fmt"

func main() {
    var a int = 10
    var b int64 = 20

    fmt.Println(a + b)
}
```

```text
编译错误：invalid operation: a + b (mismatched types int and int64)
```

必须显式转换到同一类型再算：

```go
fmt.Println(int64(a) + b)
```

```text
30
```

为什么这么较真？因为**显式转换是让你自己承担"转换是否合理"的责任**——写下 `int64(a)` 的那一刻，丢不丢精度、溢不溢出，是你看过并认可的，而不是编译器背着你做的决定。JS 的隐式转换省了打字，代价是把这些决定藏进了运行时。

---

## 2. 转换可能丢失信息

### 🕳️ 坑：转换不做安全检查，装不下就静默截断

以为会怎样：`byte(300)` 要么报错，要么"尽量"保住 300。
实际怎样：得到 `44`，无声无息。
为什么：`byte` 只有 8 位（0-255），300 的二进制是 `100101100`（9 位），转换时高位直接被剁掉，剩下 `00101100` = 44。

```go
func main() {
    var n int64 = 300
    var b byte = byte(n)

    fmt.Println(b)
}
```

```text
44
```

**一句话总结：类型转换是"裁剪装箱"，不是"安全搬运"——往小类型转之前，自己确认值在范围内。**

---

## 3. 浮点数和整数转换

浮点转整数**直接截断小数**，不是四舍五入：

```go
func main() {
    f := 3.9
    n := int(f)

    fmt.Println(n)
}
```

```text
3
```

`3.9` 变成 `3`，`-3.9` 会变成 `-3`（向零截断）。想四舍五入，用 `math.Round` 之后再转。行为等同于 JS 的 `Math.trunc(3.9)`，而不是 `parseInt` 或 `Math.round`。

---

## 4. 数字和字符串转换

"数字 123 ↔ 文本 `"123"`" 这种转换，**不能**用 `string(n)` 语法，要用标准库 `strconv`（string convert）：

| 方向 | JS | Go |
|------|-----|-----|
| 数字 → 字符串 | `String(123)` | `strconv.Itoa(123)` |
| 字符串 → 数字 | `Number("456")` | `strconv.Atoi("456")` |

（Itoa = **I**nteger **to** **A**SCII，Atoi 反之，C 语言时代传下来的老名字。）

```go
package main

import (
    "fmt"
    "strconv"
)

func main() {
    n := 123
    s := strconv.Itoa(n)
    fmt.Printf("%T %q\n", s, s)

    parsed, err := strconv.Atoi("456")
    if err != nil {
        fmt.Println("parse error:", err)
        return
    }
    fmt.Printf("%T %v\n", parsed, parsed)
}
```

```text
string "123"
int 456
```

`Atoi` 返回两个值：结果和 `err`。解析可能失败（用户输入 `"abc"` 呢），Go 强迫你处理这种情况——不像 JS 的 `Number("abc")` 给你个 `NaN` 让它继续在系统里漂流。喂给它坏数据试试：

```go
parsed, err := strconv.Atoi("abc")
fmt.Println(parsed, err)
```

```text
0 strconv.Atoi: parsing "abc": invalid syntax
```

### 🕳️ 坑：`string(65)` 不是 `"65"`

以为会怎样：像 JS 的 `String(65)` 一样得到 `"65"`。
实际怎样：得到 `"A"`。
为什么：对数字用 `string(...)` 语法，Go 的理解是"把这个数当 **Unicode 码点**，给我那个字符"——65 是字母 A 的码点。这正是 02-05 里 rune 和字符串的关系。

```go
func main() {
    fmt.Println(string(rune(65)))
    fmt.Println(strconv.Itoa(65))
}
```

```text
A
65
```

**记法：`string(n)` 问的是"码点 n 长什么样"，`strconv.Itoa(n)` 问的是"数字 n 的十进制写法"。** 想要 `"65"`，永远用 `strconv`。（新版 Go 的工具链见到直接写 `string(65)` 会警告，让你写成 `string(rune(65))` 表明"我真的要按码点转"。）

## 5. bool 不能和数字互转

JS 里 `Number(true)` 得 1、`!!0` 得 false，布尔和数字随便串门。Go 把这扇门焊死了——`bool` 和数字之间**没有任何转换**，连显式的都不行：

```go
func main() {
    ok := true
    n := int(ok)
    fmt.Println(n)
}
```

```text
编译错误：cannot convert ok (variable of type bool) to type int
```

需要"true 记 1、false 记 0"，老老实实写条件：

```go
func boolToInt(ok bool) int {
    if ok {
        return 1
    }
    return 0
}
```

这个设计和下一篇的"if 条件必须是 bool"是同一条哲学：**布尔就是布尔，数字就是数字，永不混淆。**

---

## 本篇重点

- [ ] Go 零隐式转换：`int` 和 `int64` 都不能直接混算，必须 `目标类型(值)` 显式转换。
- [ ] 往小类型转会静默截断（`byte(300)` = 44）；浮点转整数直接砍小数（`int(3.9)` = 3），不四舍五入。
- [ ] 数字 ↔ 十进制文本用 `strconv.Itoa` / `strconv.Atoi`，且 `Atoi` 的 err 必须处理。
- [ ] `string(65)` 是按码点转出 `"A"`，不是 `"65"`——JS 的 `String(65)` 直觉在这里会坑你。
- [ ] `bool` 和数字之间没有转换通道，需要 0/1 就写 if。
