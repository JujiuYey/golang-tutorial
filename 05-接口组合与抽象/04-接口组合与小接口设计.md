# 04: 接口组合与小接口设计

接口组合让多个小接口组成更大的接口。Go 更鼓励小接口，因为小接口更容易实现、替换和测试。

---

## 1. 接口嵌入

```go
type Reader interface {
    Read(p []byte) (n int, err error)
}

type Writer interface {
    Write(p []byte) (n int, err error)
}

type ReadWriter interface {
    Reader
    Writer
}
```

`ReadWriter` 同时要求 `Read` 和 `Write`。

---

## 2. 标准库里的组合

```go
type Closer interface {
    Close() error
}

type ReadCloser interface {
    Reader
    Closer
}
```

这类接口把能力拆得很细，再按需要组合。

好处是函数可以精确声明自己需要什么：

```go
func ReadAll(r Reader) ([]byte, error) {
    // 只需要 Read
}

func ReadAndClose(rc ReadCloser) ([]byte, error) {
    defer rc.Close()
    // 需要 Read 和 Close
}
```

---

## 3. 接口越小，适配越容易

```go
type UserFinder interface {
    FindUser(id int64) (User, error)
}
```

这个接口只有一个方法。测试时很容易写替身实现：

```go
type fakeUserFinder struct {
    user User
}

func (f fakeUserFinder) FindUser(id int64) (User, error) {
    return f.user, nil
}
```

如果接口包含十几个方法，测试替身也要实现十几个方法，使用成本会迅速变高。

---

## 4. 接口由调用方定义

如果函数只需要保存用户：

```go
type UserSaver interface {
    SaveUser(user User) error
}

func Register(saver UserSaver, user User) error {
    return saver.SaveUser(user)
}
```

不要强迫调用方依赖完整仓储接口：

```go
type UserRepository interface {
    FindUser(id int64) (User, error)
    SaveUser(user User) error
    DeleteUser(id int64) error
    ListUsers() ([]User, error)
}
```

`Register` 只需要 `SaveUser`，传入大接口会扩大依赖面。

---

## 5. 返回具体类型，接收接口

常见习惯：

```go
func NewMemoryStore() *MemoryStore {
    return &MemoryStore{
        data: make(map[string]string),
    }
}

func UseStore(store Store) {
    // ...
}
```

构造函数通常返回具体类型，让调用方保留完整能力。函数参数可以接收接口，只声明自己需要的能力。

---

## 6. 接口命名

单方法接口常用 `-er` 命名：

```go
type Reader interface { Read([]byte) (int, error) }
type Writer interface { Write([]byte) (int, error) }
type Closer interface { Close() error }
```

业务接口可以按能力命名：

```go
type UserFinder interface { FindUser(int64) (User, error) }
type TokenSigner interface { Sign(Token) (string, error) }
```

名字应该表达能力，不要只表达实现细节。

---

## 7. 接口隔离外部依赖

```go
type Mailer interface {
    Send(to string, subject string, body string) error
}

func SendWelcome(m Mailer, user User) error {
    return m.Send(user.Email, "welcome", "hello")
}
```

`SendWelcome` 不需要知道邮件服务的具体 SDK。测试时可以传入 fake mailer。

---

## 8. 接口和错误

标准库的 `error` 就是接口：

```go
type error interface {
    Error() string
}
```

任何实现 `Error() string` 的类型都可以作为 error 返回。

```go
type ValidationError struct {
    Field string
    Msg   string
}

func (e ValidationError) Error() string {
    return e.Field + ": " + e.Msg
}
```

---

## 9. 接口不是字段清单

接口只能描述方法，不能要求字段。

```go
type Named interface {
    Name() string
}
```

如果你想表达“有 Name 字段”，在 Go 里通常写成“有 Name 方法”，或者直接使用具体 struct。

---

## 10. 接口设计检查

写接口前问：

1. 调用方真正需要哪些方法。
2. 是否已经有两个以上实现，或者明确需要替换实现。
3. 这个接口能不能由使用方定义。
4. 测试替身实现起来是否轻松。
5. 名字是否表达行为。

这些问题能避免很多“接口看起来很完整，实际很难用”的设计。

## 练习

1. 定义 `Reader`、`Writer`、`ReadWriter`，用嵌入组合接口。
2. 把一个大 `UserRepository` 拆成 `UserFinder` 和 `UserSaver`。
3. 写函数 `SendWelcome(m Mailer, user User) error`，并写一个 fake mailer。
4. 定义一个自定义错误类型，让它满足 `error`。
5. 观察一个你写过的接口，判断它是否可以变小。
