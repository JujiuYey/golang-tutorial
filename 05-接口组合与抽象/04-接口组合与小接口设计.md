# 04: 接口组合与小接口设计

这篇解决"接口该定义多大、放在哪"的问题。学完你能看懂标准库里 `ReadWriter`、`ReadCloser` 这类名字的构成规律,并且写出"调用方只依赖自己用得到的方法"的小接口——这是 Go 代码好测试、好替换的核心手法。

一句话立场：Java/TS 的习惯是先设计一个大而全的接口再让类去实现;Go 反过来,**先有一堆小能力,需要时拼装**。

---

## 1. 接口嵌入：把小清单拼成大清单

接口里可以直接写另一个接口的名字,意思是"把它的方法清单抄过来"（接口嵌入）：

```go
type Reader interface {
    Read(p []byte) (n int, err error)
}

type Writer interface {
    Write(p []byte) (n int, err error)
}

type ReadWriter interface {
    Reader   // ← 嵌入:等于把 Read 的签名抄到这里
    Writer   // ← 等于把 Write 的签名抄到这里
}
```

`ReadWriter` 就等价于手写 `Read` + `Write` 两个签名的接口。要满足它,类型得两个方法都有。

对比 TS：类似 `interface ReadWriter extends Reader, Writer {}`,但 Go 里连 `extends` 都不用写,直接列名字。

---

## 2. 标准库里的组合

标准库把"能力"拆到不能再拆,再按需组合。跑一个完整例子——`File` 同时有 `Read` 和 `Close`,自动满足拼出来的 `ReadCloser`：

```go
package main

import "fmt"

type Reader interface {
    Read(p []byte) (n int, err error)
}

type Closer interface {
    Close() error
}

type ReadCloser interface {
    Reader
    Closer
}

type File struct{ name string }

func (f File) Read(p []byte) (int, error) {
    fmt.Println("read from", f.name)
    return 0, nil
}

func (f File) Close() error {
    fmt.Println("close", f.name)
    return nil
}

func main() {
    var rc ReadCloser = File{name: "a.txt"}
    rc.Read(nil)
    rc.Close()
}
```

```text
read from a.txt
close a.txt
```

好处是函数可以**精确声明**自己要什么,不多要一点：

```go
func ReadAll(r Reader) ([]byte, error) {
    // 只需要 Read,就只要 Reader
}

func ReadAndClose(rc ReadCloser) ([]byte, error) {
    defer rc.Close()
    // 需要 Read 和 Close,才要 ReadCloser
}
```

标准库的真实命名就是这么机械地拼的：`io.ReadWriter`、`io.ReadCloser`、`io.ReadWriteCloser`——看名字就知道方法清单。

**一句话总结：小接口是积木,大接口是拼出来的,不是设计出来的。**

---

## 3. 接口越小,适配越容易

为什么执着于小?最直接的收益在测试。一个单方法接口：

```go
type UserFinder interface {
    FindUser(id int64) (User, error)
}
```

测试时写替身(fake)只要几行：

```go
type fakeUserFinder struct {
    user User
}

func (f fakeUserFinder) FindUser(id int64) (User, error) {
    return f.user, nil
}
```

如果接口有十几个方法,替身就得实现十几个——哪怕测试只用到一个。**接口的方法数,就是每个实现者要交的税。**

---

## 4. 接口由调用方定义

上一篇的原则在这里落地。假设函数只需要"保存用户"这一个能力：

```go
type UserSaver interface {
    SaveUser(user User) error
}

func Register(saver UserSaver, user User) error {
    return saver.SaveUser(user)
}
```

反面写法是强迫调用方依赖完整仓储接口：

```go
type UserRepository interface {
    FindUser(id int64) (User, error)
    SaveUser(user User) error
    DeleteUser(id int64) error
    ListUsers() ([]User, error)
}
```

`Register` 只用 `SaveUser`,却声明依赖四个方法——测试替身要写四个方法,读代码的人也会误以为 `Register` 可能删用户。**依赖面越小,谎报的能力越少。**

多亏隐式实现,真正的数据库仓储什么都不用改:它有 `SaveUser`,就自动满足 `UserSaver`。

---

## 5. 返回具体类型,接收接口

Go 社区的一句口诀："Accept interfaces, return structs"（收接口,还具体）：

```go
// 构造函数:返回具体类型,调用方保留完整能力
func NewMemoryStore() *MemoryStore {
    return &MemoryStore{
        data: make(map[string]string),
    }
}

// 使用方:参数收接口,只声明需要的能力
func UseStore(store Store) {
    // ...
}
```

为什么不反过来?构造函数如果返回接口,调用方就被"降级"了——拿到的东西只剩接口清单上那几个方法,具体类型的其他能力全被藏起来。收窄的决定权应该留给使用的那一方。

---

## 6. 接口命名

单方法接口的惯例是**方法名 + er**：

```go
type Reader interface { Read([]byte) (int, error) }
type Writer interface { Write([]byte) (int, error) }
type Closer interface { Close() error }
```

`Read` 的人叫 Reader——像英语构词一样机械。业务接口按能力命名：

```go
type UserFinder interface { FindUser(int64) (User, error) }
type TokenSigner interface { Sign(Token) (string, error) }
```

**名字表达"能做什么",不要表达实现细节**（叫 `MySQLUserTable` 就输了——换成 Redis 存储它照样满足接口）。

---

## 7. 接口隔离外部依赖

小接口最实用的场景之一：把第三方 SDK 挡在门外。

```go
package main

import "fmt"

type User struct {
    Email string
}

type Mailer interface {
    Send(to string, subject string, body string) error
}

func SendWelcome(m Mailer, user User) error {
    return m.Send(user.Email, "welcome", "hello")
}

type fakeMailer struct {
    sent []string
}

func (f *fakeMailer) Send(to, subject, body string) error {
    f.sent = append(f.sent, to)
    return nil
}

func main() {
    fake := &fakeMailer{}
    SendWelcome(fake, User{Email: "a@example.com"})
    fmt.Println(fake.sent)
}
```

```text
[a@example.com]
```

`SendWelcome` 完全不知道邮件服务的 SDK 长什么样;测试时塞一个记录发送对象的 fake 进去,不用真发邮件就能验证逻辑。上线时换成包着真 SDK 的实现,一行业务代码不改。

## 8. 接口和错误

你天天在用的 `error`,本身就是一个单方法小接口——这也是它能被任何自定义类型"冒充"的原因：

```go
type error interface {
    Error() string
}
```

自定义错误类型,只要有 `Error() string` 就能当 error 返回：

```go
package main

import "fmt"

type ValidationError struct {
    Field string
    Msg   string
}

func (e ValidationError) Error() string {
    return e.Field + ": " + e.Msg
}

func check(age int) error {
    if age < 0 {
        return ValidationError{Field: "age", Msg: "不能为负数"}
    }
    return nil
}

func main() {
    err := check(-1)
    fmt.Println(err)
}
```

```text
age: 不能为负数
```

## 9. 接口不是字段清单

接口只能列方法,不能要求"必须有某个字段"：

```go
type Named interface {
    Name() string   // 只能这样:要求一个方法
    // Name string  // 没有这种语法
}
```

对比 TS：TS 的 interface 可以写 `name: string`,Go 不行。想表达"有 Name 数据",要么包一层取值方法 `Name() string`,要么干脆用具体 struct——**接口描述行为,数据交给 struct**。

---

## 10. 接口设计检查

写接口前过一遍这五问：

1. 调用方**真正**需要哪些方法?(多一个都别写)
2. 是否已有两个以上实现,或明确要替换实现?(没有就先用具体类型)
3. 这个接口能不能由使用方定义?(而不是实现方"出口"一个大接口)
4. 测试替身实现起来轻松吗?(方法数 = 替身成本)
5. 名字表达行为吗?(-er 命名,不泄露实现)

**一句话总结：接口是给调用方用的需求清单,不是给实现方立的功德碑。**

---

## 本篇重点

- [ ] 接口嵌入 = 把小接口的方法清单抄进大接口;标准库的 `ReadWriter`/`ReadCloser` 都是这么拼的。
- [ ] 接口方法数是实现者和测试替身的成本,越小越好;单方法接口用"方法名+er"命名。
- [ ] 接口由调用方定义,只列自己用得到的方法;隐式实现让实现方零成本适配。
- [ ] 口诀:收接口,还具体(Accept interfaces, return structs)。
- [ ] `error` 就是单方法接口;接口只能描述方法,不能要求字段。
