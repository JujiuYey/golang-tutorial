# 05: 结构体组合与嵌入

Go 没有 class,也没有 `extends`。这篇解决"没有继承怎么复用代码"的问题:答案是**组合**——把能力当零件装进结构体;嵌入(embedding)则是让零件的方法"直通"外层的语法糖。学完你能分清"嵌入很像继承但不是",并躲开同名歧义和 nil 接口字段两个坑。

---

## 1. 普通组合：把能力装成字段

最朴素的复用:想要日志能力,就装一个 `Logger` 字段,用的时候通过字段调：

```go
type Logger struct{}

func (Logger) Log(msg string) {
    fmt.Println(msg)
}

type Service struct {
    logger Logger   // 有名字的普通字段
}

func (s Service) Start() {
    s.logger.Log("start")   // 经过字段名访问
}
```

和 JS 里"对象属性挂一个工具对象"(`this.logger.log(...)`)一模一样,没有任何新概念。

---

## 2. 嵌入字段：省掉字段名,方法直通

把字段名去掉、只写类型名,就是**嵌入**：

```go
type Service struct {
    Logger        // ← 只有类型,没有字段名:嵌入字段
}
```

拆开看它带来了什么：

```text
type Service struct {
    Logger
//  ──┬───
//    └ 没写字段名 → 字段名默认就是类型名"Logger"
}
```

实跑验证——两条调用路径都通：

```go
package main

import "fmt"

type Logger struct{}

func (Logger) Log(msg string) {
    fmt.Println("[log]", msg)
}

type Service struct {
    Logger
}

func main() {
    s := Service{Logger: Logger{}}
    s.Logger.Log("走完整路径")   // 完整路径:字段名就是类型名
    s.Log("走提升方法")           // 短路径:方法被"提升"到外层
}
```

```text
[log] 走完整路径
[log] 走提升方法
```

`s.Log(...)` 能直接调,是因为嵌入字段的方法被**提升**(promoted)到了外层——Go 自动帮你补上了 `.Logger` 那一跳。

**一句话总结：嵌入 = 匿名字段 + 方法直通,本质还是"有个字段"。**

---

## 3. 🕳️ 坑：嵌入不是继承

以为会怎样：`Service` 嵌入了 `Logger`,那 `Service` 就是一种 `Logger`,能传给要 `Logger` 的函数(继承思维:子类 is-a 父类)。
实际怎样：编译错误。

```go
func UseLogger(l Logger) {}

func main() {
    s := Service{Logger: Logger{}}
    UseLogger(s)
}
```

```text
# mod05/a05_notinherit
./main.go:19:12: cannot use s (variable of struct type Service) as Logger value in argument to UseLogger
```

为什么：嵌入只是"把 Logger 装进 Service 里"(has-a),不是"Service 是一种 Logger"(is-a)。想传,把里面那个零件掏出来传：

```go
UseLogger(s.Logger)   // 编译通过
```

| | 继承(Java/TS class) | 嵌入(Go) |
|---|---|---|
| 关系 | is-a(是一种) | has-a(有一个) |
| 子类型转换 | 子类可当父类传参 | ❌ 外层不能当内层传参 |
| 方法复用 | 继承获得 | 提升获得(效果相似) |

**一句话总结：嵌入借的是方法,不借身份。**

---

## 4. 嵌入接口：注入能力的插座

嵌入的不一定是 struct,也可以是**接口**——相当于给结构体留一个"能力插座",创建时插入任意实现：

```go
type Logger interface {
    Log(msg string)
}

type Service struct {
    Logger   // 嵌入接口:插座
}
```

实跑——插上实现就能用,忘了插就是 nil 接口,一调就 panic：

```go
type ConsoleLogger struct{}

func (ConsoleLogger) Log(msg string) {
    fmt.Println("[console]", msg)
}

func main() {
    ok := Service{Logger: ConsoleLogger{}}
    ok.Log("start")          // 插了 ConsoleLogger

    var bad Service          // 零值:Logger 字段是 nil 接口
    bad.Log("start")
}
```

```text
[console] start
panic: runtime error: invalid memory address or nil pointer dereference
[signal SIGSEGV: segmentation violation code=0x2 addr=0x0 pc=0x104edd484]

goroutine 1 [running]:
main.main()
	/.../main.go:24 +0x34
exit status 2
```

🕳️ 坑：嵌入接口的结构体,**零值不可用**——接口字段没人赋值就是 nil(上上篇的 nil 接口),调方法直接 panic。用这种模式要保证构造时必填,或者在方法里判空。

## 5. 匿名字段的可见性：字段也会提升

提升的不只方法,**字段**也一样：

```go
package main

import (
    "fmt"
    "time"
)

type Config struct {
    Timeout time.Duration
}

type Client struct {
    Config
}

func main() {
    c := Client{Config: Config{Timeout: time.Second}}
    fmt.Println(c.Timeout)          // 短路径
    fmt.Println(c.Config.Timeout)   // 完整路径
}
```

```text
1s
1s
```

`c.Timeout` 和 `c.Config.Timeout` 是同一个字段。多个嵌入字段撞名时会怎样?——下一节的坑。

## 6. 方法提升与覆盖、同名歧义

### 外层同名方法"遮住"提升方法

外层类型自己定义了同名方法时,短路径优先走外层——想调里层的,写完整路径：

```go
package main

import "fmt"

type Logger struct{}

func (Logger) Name() string {
    return "logger"
}

type Service struct {
    Logger
}

func (Service) Name() string {
    return "service"
}

func main() {
    s := Service{}
    fmt.Println(s.Name())          // 外层优先
    fmt.Println(s.Logger.Name())   // 指名道姓调里层
}
```

```text
service
logger
```

看着像 JS 类的方法覆写,但注意:没有 `super`,里层方法也**不会**反过来调到外层——两个方法互不知道对方存在,只是短路径的查找顺序是"外层先"。

### 🕳️ 坑：两个嵌入字段撞名,短路径直接报错

```go
type Logger struct{}
func (Logger) Name() string { return "logger" }

type Tracer struct{}
func (Tracer) Name() string { return "tracer" }

type Service struct {
    Logger
    Tracer
}

func main() {
    s := Service{}
    fmt.Println(s.Name())
}
```

```text
# mod05/a05_ambig
./main.go:20:16: ambiguous selector s.Name
```

Go 不搞"谁先声明听谁的"这种潜规则——歧义就编译失败,逼你写完整路径 `s.Logger.Name()` 或 `s.Tracer.Name()`。

**一句话总结：短路径只是语法糖,撞名时糖就失效,回到完整路径。**

## 7. 组合接口能力

实战里最常见的形态:结构体用**普通字段**装几个接口,每个接口是一项可替换的能力：

```go
type Handler struct {
    Store  UserStore   // 能力1:查用户(接口)
    Logger Logger      // 能力2:写日志(接口)
}

func (h Handler) Serve(id int64) error {
    user, err := h.Store.FindUser(id)
    if err != nil {
        return err
    }
    h.Logger.Log("found " + user.Name)
    return nil
}
```

`Handler` = 查用户 + 写日志,两个零件各自独立替换:测试时插 fake store,上线时插真数据库,`Serve` 一行不改——这就是上一篇"小接口 + 依赖注入"和这一篇"组合"的合流。

---

## 8. 组合优先于继承式思维

带着 class 背景写 Go,最需要扭转的一个问题句式：

- ❌ 继承思维:"这个类型**是什么**?"(Server 是不是一种 Database?)
- ✅ 组合思维:"这个类型**需要哪些能力**?"(Server 需要一个能查数据的 Store)

```go
type Server struct {
    Store Store   // 需要什么,装什么
}
```

不需要设计类型树、不需要纠结"该继承谁"——列出能力清单,装进来,完事。

---

## 9. 嵌入的使用边界

嵌入(短路径)和普通字段(完整路径)怎么选:

| 用嵌入 | 用普通字段 |
|---|---|
| 里层方法确实该成为外层的公开 API | 只是内部实现细节 |
| 外层和这项能力关系稳定 | 不想把内部能力暴露出去 |
| 提升能省大量转发样板 | 嵌入后方法来源会变模糊 |

拿不准就用普通字段——多打一个字段名不亏,误暴露 API 才亏。

---

## 10. 组合和接口的关系

收个尾,把 04、05 两篇拼起来：

```go
type Service struct {
    Store  Store    // 组合:获得能力
    Logger Logger
}

func NewService(store Store, logger Logger) Service {
    return Service{Store: store, Logger: logger}
}
```

- **组合**回答"具体类型怎么攒出来"——把能力装成字段。
- **接口**回答"调用边界长什么样"——字段的类型是小接口,实现可换。

两者一个管内、一个管外,合起来就是 Go 替代继承的完整方案。

---

## 本篇重点

- [ ] Go 用组合替代继承:能力装成字段;嵌入(只写类型不写字段名)让方法和字段"提升"到外层。
- [ ] 嵌入不是继承:has-a 不是 is-a,外层类型不能当里层类型传参。
- [ ] 外层同名方法遮住提升方法(没有 super);两个嵌入字段撞名,短路径编译报错 `ambiguous selector`。
- [ ] 嵌入接口是能力插座,但零值时接口字段是 nil,调用即 panic——构造时必须插上实现。
- [ ] 拿不准嵌入还是普通字段时选普通字段;问"需要哪些能力",不问"是什么"。
