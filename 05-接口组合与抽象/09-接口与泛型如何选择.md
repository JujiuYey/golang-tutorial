# 09: 接口与泛型如何选择

本模块学了两种抽象工具,这篇解决"遇到具体问题该掏哪个"的选择题。学完你能用一句口诀 + 一张判断表快速定夺,还能识别"选错了"的典型信号。

先把口诀立在这:**接口抽象"行为"(它会做什么),泛型抽象"类型"(同一段逻辑套不同类型)。** 下面逐条展开。

---

## 1. 接口适合行为边界

回看接口的典型用法：

```go
package main

import (
    "bytes"
    "fmt"
)

type Writer interface {
    Write(p []byte) (int, error)
}

func Save(w Writer, data []byte) error {
    _, err := w.Write(data)
    return err
}

func main() {
    var buf bytes.Buffer
    Save(&buf, []byte("hello"))
    fmt.Println(buf.String())
}
```

```text
hello
```

`Save` 的诉求是"给我个能写的东西"——文件、网络连接、内存缓冲,**运行时**装谁都行。这类问题的特征:

- 需要替换实现(测试塞 fake,上线换真的)。
- 需要隔离外部依赖(邮件 SDK、数据库)。
- 需要运行时动态分派(同一行代码,不同对象不同行为)。
- 只关心"它会做什么",不关心"它是什么"。

---

## 2. 泛型适合同形算法

再看泛型的典型用法：

```go
package main

import "fmt"

func Filter[T any](items []T, keep func(T) bool) []T {
    out := make([]T, 0, len(items))
    for _, item := range items {
        if keep(item) {
            out = append(out, item)
        }
    }
    return out
}

func main() {
    evens := Filter([]int{1, 2, 3, 4}, func(n int) bool { return n%2 == 0 })
    fmt.Println(evens)
    fmt.Printf("%T\n", evens)
}
```

```text
[2 4]
[]int
```

`Filter` 的诉求是"逻辑一份,类型随便,**但进什么类型出什么类型**"——`%T` 打出 `[]int`,类型关系在编译期就锁死了。这类问题的特征:

- 算法对所有类型一模一样(查找、过滤、映射、容器)。
- 输入输出之间有类型关系要保住(`[]T → []T`)。
- 不想把数据降级成 `any` 再断言回来。

| | 接口 | 泛型 |
|---|---|---|
| 抽象的是 | 行为(方法) | 类型(数据) |
| 决定时机 | 运行时(动态分派) | 编译期(实例化) |
| JS 里的近亲 | 鸭子类型 | 函数天生不挑类型 |
| 典型场景 | 依赖注入、可替换实现 | 容器、Map/Filter 类算法 |

---

## 3. 具体类型优先

选择题还有个隐藏选项,而且是**默认答案**:两个都不用。

```go
// 只有 User 会被校验?那就写 User 的校验
func ValidateUser(user User) error {
    // ...
}
```

不要一上来就:

```go
func Validate[T any](v T) error {   // 校验"任意类型"?约束是 any,函数体啥也干不了
    // ...
}
```

**抽象要等重复真的出现**(而且是稳定重复)再做——想象中的复用需求,九成不会来,先付出的复杂度却是实打实的。

---

## 4. any 通常不是更好的抽象

一个函数如果长这样：

```go
func Save(v any) error {
    switch v.(type) {
    case User:  // ...
    case Order: // ...
    }
}
```

内部靠 type switch 撑着,说明类型信息在门口就丢了、进门再费劲捡回来。三条更好的出路:

- 各分支行为一致 → 抽成**接口**方法(`v.Save()`)。
- 逻辑同形只差类型 → 用**泛型**保住类型。
- 各分支各干各的 → 老实拆成 `SaveUser`、`SaveOrder`。

**一句话总结：函数体里 type switch 连篇,是选错抽象的红灯。**

---

## 5. 接口和泛型可以结合

不是二选一——泛型管数据类型,接口管行为,经常一起上：

```go
package main

import (
    "errors"
    "fmt"
)

type Validator[T any] interface {
    Validate(T) error
}

func ValidateAll[T any](items []T, validator Validator[T]) error {
    for _, item := range items {
        if err := validator.Validate(item); err != nil {
            return err
        }
    }
    return nil
}

type AgeChecker struct{}

func (AgeChecker) Validate(age int) error {
    if age < 0 {
        return errors.New("年龄不能为负数")
    }
    return nil
}

func main() {
    fmt.Println(ValidateAll([]int{18, 30}, AgeChecker{}))
    fmt.Println(ValidateAll([]int{18, -1}, AgeChecker{}))
}
```

```text
<nil>
年龄不能为负数
```

分工看得很清楚:泛型 `T` 保证 items 的元素类型和校验器匹配(传 `[]int` 就得配 int 的校验器,编译期查),接口 `Validator` 让校验规则可以随便换。

---

## 6. 泛型约束里的接口不是普通接口

上一篇的知识,放进选择视角再对一次表——同一个 `interface` 关键字,两种身份别混:

```go
type Ordered interface {          // 约束接口:类型集合
    ~int | ~int64 | ~float64 | ~string
}

type Reader interface {           // 普通接口:方法集合
    Read([]byte) (int, error)
}
```

| | 描述什么 | 能声明变量? | 用在哪 |
|---|---|---|---|
| 普通接口 | 方法集合 | ✅ | 参数、字段、返回值 |
| 约束接口 | 类型集合 | ❌ | 只能进方括号 |

看到 `interface` 先看里面:列方法的是行为抽象,列类型的是泛型约束。

---

## 7. 性能不是唯一理由

技术上:接口调用走动态分派(运行时查表),泛型让编译器保留更多类型信息、可能生成更快的代码。于是有人想把接口全改泛型"提速"。

别。业务代码里这点差异几乎永远不是瓶颈,**可读性和边界清晰值钱得多**。先问抽象对不对,再谈快不快——方向错了,快也是白快。

---

## 8. 错误选择的信号

代码写出这些味道,说明该回头换工具了:

**接口用错的信号:**

- 接口很大,每个实现都很痛苦。
- 只有一个实现,调用方也不需要替换。
- 接口定义在实现方的包里,所有人被迫依赖它。
- 方法是 CRUD 全家桶,调用方实际只用一两个(→ 拆小,见第 04 篇)。

**泛型用错的信号:**

- 类型参数一大把,签名没法读(→ 第 08 篇的可读性红线)。
- 函数体里大量 type switch(约束没圈对,类型信息还是丢了)。
- 为业务概念硬凑类型集合("金额|库存|年龄",第 07 篇警告过)。
- 只有一个具体类型在用(→ 退回具体代码)。

---

## 9. 一个简单判断表

| 问题 | 优先选择 |
| --- | --- |
| 只需要调用一个行为 | 小接口 |
| 同一算法用于不同元素类型 | 泛型 |
| 只有一个具体类型 | 具体类型 |
| 需要运行时切换实现 | 接口 |
| 需要保留 `[]T → []T` 类型关系 | 泛型 |
| 只是调试输出 | `any` 或 `fmt` |
| 外部依赖需要替身测试 | 接口 |

**速记:换"实现"用接口,换"类型"用泛型,不用换就用具体类型。**

---

## 10. 设计顺序

实践中的思考流水线:

1. 先写具体代码,把业务规则写清楚。
2. 看到**两处以上稳定重复**,才考虑抽象。
3. 重复的是行为边界 → 接口。
4. 重复的是算法形状 → 泛型。
5. 抽完发现调用方更难读了 → 撤回,用具体代码。

第 5 步不是失败,是设计的正常回路。**好的抽象让调用方少知道一点,而不是让读者多猜一层。**

---

## 本篇重点

- [ ] 口诀:接口抽象行为(运行时可换实现),泛型抽象类型(编译期保类型关系),默认答案是具体类型。
- [ ] 函数体 type switch 连篇 = 选错抽象的红灯,改用接口方法、泛型或拆函数。
- [ ] 两者可结合:`ValidateAll[T](items []T, v Validator[T])`——泛型锁类型,接口换行为。
- [ ] `interface` 两副面孔:列方法的能当变量类型,列类型集合的只能进方括号。
- [ ] 设计顺序:具体代码 → 等稳定重复 → 按重复种类选工具 → 更难读就撤回。
