# 09: 接口与泛型如何选择

接口和泛型都能做抽象，但抽象的方向不同。接口关注行为，泛型关注类型参数上的通用代码。

---

## 1. 接口适合行为边界

```go
type Writer interface {
    Write(p []byte) (int, error)
}

func Save(w Writer, data []byte) error {
    _, err := w.Write(data)
    return err
}
```

`Save` 不关心 `w` 的具体类型，只关心它能写入数据。

适合接口的问题：

- 需要替换实现。
- 需要隔离外部依赖。
- 需要运行时动态分派。
- 只关心对象能做什么。

---

## 2. 泛型适合同形算法

```go
func Filter[T any](items []T, keep func(T) bool) []T {
    out := make([]T, 0, len(items))
    for _, item := range items {
        if keep(item) {
            out = append(out, item)
        }
    }
    return out
}
```

`Filter` 的逻辑和元素类型无关，而且返回值要保留元素类型。

适合泛型的问题：

- 算法对多种类型都一样。
- 输入输出之间有明确类型关系。
- 不想把具体类型丢成 `any`。
- 容器、集合、查找、转换、比较等通用逻辑。

---

## 3. 具体类型优先

如果只有一个类型会用到，先写具体代码。

```go
func ValidateUser(user User) error {
    // ...
}
```

不要一开始就写：

```go
func Validate[T any](v T) error {
    // ...
}
```

泛型应该来自稳定重复，而不是来自想象中的复用。

---

## 4. any 通常不是更好的抽象

```go
func Save(v any) error {
    // type switch...
}
```

如果函数内部要不断 type switch，说明类型信息已经丢失。很多时候更好的选择是：

- 使用接口表达需要的方法。
- 使用泛型保留类型关系。
- 拆成多个具体函数。

---

## 5. 接口和泛型可以结合

```go
type Validator[T any] interface {
    Validate(T) error
}

func ValidateAll[T any](items []T, validator Validator[T]) error {
    for _, item := range items {
        if err := validator.Validate(item); err != nil {
            return err
        }
    }
    return nil
}
```

这里泛型保留 `items` 的元素类型，接口表达验证行为。

---

## 6. 泛型约束里的接口不是普通接口

```go
type Ordered interface {
    ~int | ~int64 | ~float64 | ~string
}
```

这个接口主要用于约束类型参数。它描述的是类型集合。

普通接口：

```go
type Reader interface {
    Read([]byte) (int, error)
}
```

它描述的是方法集合，可以作为变量类型。

这两个都使用 `interface` 语法，但含义不同。

---

## 7. 性能不是唯一理由

接口调用涉及动态分派，泛型通常能让编译器保留更多类型信息。但在大多数业务代码里，可读性和边界清晰更重要。

不要因为“泛型可能更快”就把所有接口改成泛型。先判断抽象是否正确。

---

## 8. 错误选择的信号

接口用错的信号：

- 接口很大，实现很痛苦。
- 只有一个实现，而且调用方不需要替换。
- 接口放在实现方包里，外部被迫依赖。
- 接口方法只是 CRUD 清单，调用方实际只用一两个。

泛型用错的信号：

- 类型参数很多，函数签名难读。
- 函数体里大量 type switch。
- 约束为了业务概念硬凑类型集合。
- 只有一个具体类型在用。

---

## 9. 一个简单判断表

| 问题 | 优先选择 |
| --- | --- |
| 只需要调用一个行为 | 小接口 |
| 同一算法用于不同元素类型 | 泛型 |
| 只有一个具体类型 | 具体类型 |
| 需要运行时切换实现 | 接口 |
| 需要保留 `[]T -> []T` 类型关系 | 泛型 |
| 只是调试输出 | `any` 或 `fmt` |
| 外部依赖需要替身测试 | 接口 |

---

## 10. 设计顺序

实践中可以按这个顺序思考：

1. 先写具体代码，把业务规则写清楚。
2. 看到两个以上稳定重复，再考虑抽象。
3. 如果重复的是行为边界，用接口。
4. 如果重复的是算法形状，用泛型。
5. 如果抽象让调用方更难读，撤回具体代码。

好的抽象应该让调用方少知道一点，而不是让读者多猜一层。

---

## 练习

1. 判断 `io.Reader` 为什么应该是接口。
2. 判断 `Filter([]User)` 和 `Filter([]Order)` 是否适合泛型。
3. 把一个 `func Print(v any)` 改成更具体的接口或泛型版本。
4. 找一个只有单实现的大接口，尝试拆成调用方需要的小接口。
5. 写一个泛型函数，再解释为什么它不应该用 `any` 实现。
