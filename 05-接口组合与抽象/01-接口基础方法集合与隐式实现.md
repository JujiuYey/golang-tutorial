# 01: 接口基础：方法集合与隐式实现

接口是一组方法签名。一个类型只要拥有接口要求的全部方法，就实现了这个接口。

---

## 1. 定义接口

```go
type Reader interface {
    Read(p []byte) (n int, err error)
}
```

接口只描述行为，不保存字段。

再看一个业务例子：

```go
type Notifier interface {
    Notify(message string) error
}
```

任何拥有 `Notify(string) error` 方法的类型，都可以作为 `Notifier` 使用。

---

## 2. 隐式实现

Go 不需要写 `implements`。

```go
type EmailNotifier struct {
    Address string
}

func (e EmailNotifier) Notify(message string) error {
    fmt.Println("send email to", e.Address, message)
    return nil
}
```

`EmailNotifier` 拥有 `Notify(string) error` 方法，所以它自动满足 `Notifier` 接口。

```go
var n Notifier = EmailNotifier{Address: "a@example.com"}
_ = n.Notify("hello")
```

---

## 3. 方法签名必须完全匹配

```go
type Notifier interface {
    Notify(message string) error
}

type BadNotifier struct{}

func (BadNotifier) Notify(message string) {
    fmt.Println(message)
}
```

`BadNotifier` 不满足 `Notifier`，因为返回值不匹配。

参数类型、参数顺序、返回值数量、返回值类型都必须一致。参数名不属于签名的一部分。

---

## 4. 接口变量

接口变量可以保存实现了该接口的具体值。

```go
type Shape interface {
    Area() float64
}

type Rectangle struct {
    Width  float64
    Height float64
}

func (r Rectangle) Area() float64 {
    return r.Width * r.Height
}

func PrintArea(s Shape) {
    fmt.Println(s.Area())
}

func main() {
    PrintArea(Rectangle{Width: 3, Height: 4})
}
```

`PrintArea` 不关心具体类型，只要求传入值能计算面积。

---

## 5. 值接收者的方法集合

```go
type Counter struct {
    N int
}

func (c Counter) Value() int {
    return c.N
}
```

如果方法使用值接收者，那么 `Counter` 和 `*Counter` 都可以调用这个方法。

```go
type Valuer interface {
    Value() int
}

var a Valuer = Counter{N: 1}
var b Valuer = &Counter{N: 2}

fmt.Println(a.Value(), b.Value())
```

---

## 6. 指针接收者的方法集合

```go
func (c *Counter) Inc() {
    c.N++
}
```

如果接口要求 `Inc`：

```go
type Incrementer interface {
    Inc()
}
```

那么 `*Counter` 满足接口，`Counter` 不满足接口。

```go
var ok Incrementer = &Counter{}

// var bad Incrementer = Counter{} // 编译错误
```

原因是 `Inc` 属于 `*Counter` 的方法集合，不属于 `Counter` 的方法集合。

---

## 7. 编译期接口检查

可以显式写一行编译期检查：

```go
var _ Notifier = EmailNotifier{}
```

如果 `EmailNotifier` 不满足 `Notifier`，这一行会编译失败。

指针接收者时：

```go
var _ Incrementer = (*Counter)(nil)
```

这种写法常用于库代码，明确表达“这个类型应该满足这个接口”。

---

## 8. 接口应该由使用方定义

接口通常放在使用它的地方。

```go
type UserStore interface {
    FindUser(id int64) (User, error)
}

func GetProfile(store UserStore, id int64) (Profile, error) {
    user, err := store.FindUser(id)
    if err != nil {
        return Profile{}, err
    }
    return Profile{Name: user.Name}, nil
}
```

`GetProfile` 只需要 `FindUser`，所以它定义小接口。具体数据库实现不需要提前声明自己实现了这个接口。

---

## 9. 小接口更灵活

Go 标准库里有很多小接口：

```go
type Reader interface {
    Read(p []byte) (n int, err error)
}

type Writer interface {
    Write(p []byte) (n int, err error)
}

type Closer interface {
    Close() error
}
```

小接口的好处是组合自由、测试容易、实现成本低。

---

## 10. 过早抽接口的问题

如果只有一个实现，调用方也没有替换需求，直接使用具体类型通常更清晰。

接口应该服务于边界：

- 需要替换实现。
- 需要隔离外部依赖。
- 需要表达一小组稳定行为。
- 需要让测试可以注入替身实现。

接口不是为了让代码看起来更“高级”。
