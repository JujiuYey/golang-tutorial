# 01: 接口基础：方法集合与隐式实现

这篇解决一个核心问题：怎么让函数"不挑具体类型，只挑能力"。学完你能看懂 Go 里无处不在的 `io.Reader`、`error` 这类接口，并躲开"指针接收者导致类型不满足接口"这个新手最常撞的编译错误。

先给 JS 用户校准坐标：你早就用过鸭子类型——"只要它会 `quack()`，我就当它是鸭子"。Go 的接口就是把这句话写成代码，**但检查从运行时提前到了编译期**：JS 里传错对象要等调用 `quack()` 那一刻才炸，Go 在编译时就告诉你"这个类型缺哪个方法"。

---

## 1. 定义接口

接口就是一张"能力清单"：列出几个方法签名，谁有这些方法，谁就算数。

```go
type Notifier interface {
    Notify(message string) error
}
```

拆开看：

```text
type Notifier interface {
//   ───┬────  ────┬────
//  ①接口名     ②关键字 interface（不是 struct）
    Notify(message string) error
//  ──────────┬─────────────────
//  ③方法签名：只有签名，没有函数体
}
```

接口只描述"会做什么"，不保存字段、不写实现。任何拥有 `Notify(string) error` 方法的类型，都可以当 `Notifier` 用。

**一句话总结：接口 = 一组方法签名 = 一张能力清单。**

---

## 2. 隐式实现：不需要写 implements

Go 没有 `implements` 关键字。类型不用"声明"自己实现了某个接口——**方法齐了就自动算数**：

```go
package main

import "fmt"

type Notifier interface {
    Notify(message string) error
}

type EmailNotifier struct {
    Address string
}

func (e EmailNotifier) Notify(message string) error {
    fmt.Println("send email to", e.Address, ":", message)
    return nil
}

func main() {
    var n Notifier = EmailNotifier{Address: "a@example.com"}
    _ = n.Notify("hello")
}
```

```text
send email to a@example.com : hello
```

注意 `EmailNotifier` 的定义里没有一个字提到 `Notifier`——它甚至可以定义在另一个包里、写在接口出现之前。只要方法对上，赋值就成立。

对比 JS/TS：TS 的 `class A implements B` 是显式声明；Go 是"长得像就是"（结构化类型）。这个设计的威力在第 8 节会看到：**接口可以事后定义，给别人写好的类型"追加"抽象**。

---

## 3. 方法签名必须完全匹配

鸭子类型不等于"差不多就行"。方法名对了但签名差一点，照样不算：

```go
type Notifier interface {
    Notify(message string) error
}

type BadNotifier struct{}

func (BadNotifier) Notify(message string) {   // 少了 error 返回值
    fmt.Println(message)
}

func main() {
    var n Notifier = BadNotifier{}
    _ = n
}
```

```text
# mod05/a01_bad
./main.go:16:19: cannot use BadNotifier{} (value of struct type BadNotifier) as Notifier value in variable declaration: BadNotifier does not implement Notifier (wrong type for method Notify)
		have Notify(string)
		want Notify(string) error
```

编译器把差异指得很清楚：`have`（你有的）和 `want`（接口要的）差一个 `error`。这就是"编译期鸭子类型"和 JS 运行时鸭子类型的区别——错误在写代码时就暴露。

要求完全一致的有：参数类型、参数顺序、返回值数量、返回值类型。**参数名不算**——`Notify(message string)` 和 `Notify(msg string)` 是同一个签名。

---

## 4. 接口变量：函数只挑能力，不挑类型

接口最常见的用法是当函数参数——"我不管你是谁，只要你会算面积"：

```go
package main

import "fmt"

type Shape interface {
    Area() float64
}

type Rectangle struct {
    Width  float64
    Height float64
}

func (r Rectangle) Area() float64 {
    return r.Width * r.Height
}

func PrintArea(s Shape) {
    fmt.Println(s.Area())
}

func main() {
    PrintArea(Rectangle{Width: 3, Height: 4})
}
```

```text
12
```

`PrintArea` 不认识 `Rectangle`，也不需要认识。以后新增 `Circle`、`Triangle`，只要有 `Area() float64` 方法，`PrintArea` 一行不用改。

**一句话总结：接口参数 = 把"要求"从'你是什么类型'降级成'你会什么方法'。**

---

## 5. 值接收者的方法集合

上一模块学过方法接收者有值和指针两种。这件事在接口这里产生一个关键规则——先看宽松的一半。

方法用**值接收者**定义时，`Counter` 和 `*Counter` 都能满足接口：

```go
package main

import "fmt"

type Counter struct {
    N int
}

func (c Counter) Value() int {   // 值接收者
    return c.N
}

type Valuer interface {
    Value() int
}

func main() {
    var a Valuer = Counter{N: 1}    // 值,OK
    var b Valuer = &Counter{N: 2}   // 指针,也 OK
    fmt.Println(a.Value(), b.Value())
}
```

```text
1 2
```

道理不难：有了指针随时能取到值，所以值接收者的方法指针也能用。

---

## 6. 指针接收者的方法集合

### 🕳️ 坑：指针接收者方法,值不满足接口

以为会怎样：`Inc` 是 `Counter` 的方法,那 `Counter` 值也该满足 `Incrementer` 吧。
实际怎样：只有 `*Counter` 满足，`Counter` 直接编译报错。

```go
type Counter struct {
    N int
}

func (c *Counter) Inc() {   // 指针接收者
    c.N++
}

type Incrementer interface {
    Inc()
}

func main() {
    var bad Incrementer = Counter{}
    _ = bad
}
```

```text
# mod05/a01_inc_bad
./main.go:16:24: cannot use Counter{} (value of struct type Counter) as Incrementer value in variable declaration: Counter does not implement Incrementer (method Inc has pointer receiver)
```

为什么：`Inc` 要通过指针改原值。而放进接口的 `Counter{}` 是一份**值的副本**，拿不到稳定的原件地址——Go 干脆规定：指针接收者的方法只属于 `*Counter` 的方法集合。改成取地址就好了：

```go
var ok Incrementer = &Counter{}   // 编译通过
```

注意报错信息里的 `method Inc has pointer receiver`——以后见到这句话,条件反射加 `&`。

| 方法接收者 | `Counter` 满足接口? | `*Counter` 满足接口? |
|-----------|:---:|:---:|
| `func (c Counter)` 值 | ✅ | ✅ |
| `func (c *Counter)` 指针 | ❌ | ✅ |

**一句话总结：有任何一个指针接收者方法,就用指针去满足接口。**

---

## 7. 编译期接口检查

隐式实现有个小代价：没有 `implements`，改错方法签名时不会立刻发现"某类型不再满足某接口"。惯用的保险丝是一行"故意的赋值"：

```go
var _ Notifier = EmailNotifier{}
```

拆开看：

```text
var _ Notifier = EmailNotifier{}
//  ┬ ───┬────   ──────┬────────
//  ①    ②             ③
// ① 空白标识符:不要这个变量,只借用赋值动作
// ② 目标接口
// ③ 被检查的类型(造一个零值)
```

如果 `EmailNotifier` 不满足 `Notifier`，这行编译失败。比如用它检查上一节的坑：

```go
var _ Incrementer = Counter{}   // Inc 是指针接收者
```

```text
# mod05/a01_check_fail
./main.go:11:21: cannot use Counter{} (value of struct type Counter) as Incrementer value in variable declaration: Counter does not implement Incrementer (method Inc has pointer receiver)
```

想检查指针类型又不想真的分配内存,写 `(*Counter)(nil)`（把 nil 转成 `*Counter` 类型）：

```go
var _ Incrementer = (*Counter)(nil)   // 编译通过
```

这种写法常见于库代码，明确表达"这个类型承诺满足这个接口"，改坏了编译器第一时间报警。

## 8. 接口应该由使用方定义

这是 Go 和 Java/TS 习惯差别最大的一点：接口不放在实现方旁边，**放在需要它的函数旁边**。

```go
type UserStore interface {
    FindUser(id int64) (User, error)
}

func GetProfile(store UserStore, id int64) (Profile, error) {
    user, err := store.FindUser(id)
    if err != nil {
        return Profile{}, err
    }
    return Profile{Name: user.Name}, nil
}
```

`GetProfile` 只用得到 `FindUser`，所以它自己定义一个单方法小接口。真正的数据库实现可能有几十个方法——不要紧，隐式实现意味着它**自动**满足这个小接口，不需要提前声明、甚至不需要知道这个接口存在。

**一句话总结：调用方列自己的需求清单,实现方不用签字。**

---

## 9. 小接口更灵活

标准库把这个哲学贯彻到底——最有名的几个接口都只有一个方法：

```go
type Reader interface {
    Read(p []byte) (n int, err error)
}

type Writer interface {
    Write(p []byte) (n int, err error)
}

type Closer interface {
    Close() error
}
```

方法越少,能满足它的类型就越多（文件、网络连接、内存缓冲、压缩流……全是 `Reader`）。小接口组合自由（下下篇细讲）、测试替身好写、实现成本低。

---

## 10. 过早抽接口的问题

接口不是写得越多越"工程化"。如果只有一个实现、调用方也没有替换需求，直接用具体类型更清晰——少一层跳转,读代码时不用猜"这背后到底是谁"。

什么时候值得抽接口——**边界处**：

- 需要替换实现（内存版 / 数据库版）。
- 需要隔离外部依赖（邮件服务、支付 SDK）。
- 需要表达一小组稳定行为（`Reader`、`error`）。
- 测试时需要注入替身实现。

**一句话总结：接口服务于边界,不是为了让代码看起来更"高级"。**

---

## 本篇重点

- [ ] 接口 = 一组方法签名。Go 是隐式实现：方法齐了就自动满足,没有 `implements`——JS 的鸭子类型,但检查在编译期。
- [ ] 签名必须完全匹配（参数、返回值的类型和顺序）,参数名不算。
- [ ] 方法集合规则：值接收者方法,值和指针都满足接口；指针接收者方法,**只有指针满足**（报错关键词 `pointer receiver`）。
- [ ] `var _ 接口 = 类型{}` 是一行编译期保险丝,库代码常用。
- [ ] 接口由使用方定义、越小越好；只有一个实现时先用具体类型。
